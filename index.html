Cloud Fundamentals
Five NIST characteristics, four deployment models, three service models, AWS global infrastructure, shared responsibility model, and cloud economics. Every architectural decision traces back here.
📚 Ch.1 — Understanding Cloud Fundamentals
🏋
Exam focus: Know all five NIST characteristics by name. Know which model AWS is (public). Know who owns what in shared responsibility — it shifts per service type. Managed service = AWS owns more.
Exam format & logistics [web-sourced]
Questions
65 questions, 130 minutes. 50 scored + 15 unscored.
Passing score
720 / 1000 (scaled). Roughly 72%.
Question types
Multiple-choice (1 of 4) and multiple-response (2–3 of 5). No penalty for guessing.
Domains
Secure (30%), Resilient (26%), High-Performing (24%), Cost-Optimized (20%) — see Ch. 12–15.
⚠
Biggest meta-trap: when several answers are technically correct, pick the one matching the qualifier — "MOST cost-effective," "LEAST operational overhead," "MOST resilient." Don't pick the most powerful/complex architecture if the question asks for the cheapest that meets requirements.
AWS Well-Architected Framework — 6 pillars [web-sourced]
Operational Excellence
Run and monitor systems; continuously improve processes.
Security
Protect data, systems, and assets; least privilege; defense in depth.
Reliability
Recover from failure; meet demand; auto-scaling and self-healing.
Performance Efficiency
Use resources efficiently; right service for the job.
Cost Optimization
Avoid unnecessary costs; pay for what you use.
Sustainability
Minimize environmental impact of cloud workloads.
Five NIST cloud characteristics
⚡
On-demand self-service
Provision resources automatically without human intervention from the provider. No tickets, no procurement delays.
🌐
Broad network access
Accessible over standard protocols from mobile, laptop, tablet — any heterogeneous client platform.
🔗
Resource pooling
Multi-tenant model. Physical and virtual resources dynamically assigned. You don't know or control the exact physical location. Economies of scale.
📈
Rapid elasticity
Scale outward and inward — sometimes automatically — matching demand. Appears unlimited to the consumer.
💰
Measured service
Usage monitored, controlled, and reported automatically. Pay-per-use. Transparency for both provider and consumer.
Deployment models
Public
Services over the internet, shared infrastructure among multiple customers. AWS is a public cloud. Strict logical isolation between tenants. Pay-as-you-go, highly scalable.
Private
Dedicated to one organization. Full control and customization. Higher cost, limited elasticity. Banks, government, military with strict compliance requirements.
Hybrid
Mix of private and public. Sensitive data on-premises, scalable workloads in public cloud. More complex but flexible. Common during cloud migration journeys.
Community
Shared among organizations with common compliance goals (e.g., government agencies). Cost and resource sharing. Rarely tested on SAA-C03.
Service models
IaaS — Infrastructure as a Service
You manage: OS, runtime, apps, data.
AWS manages: hardware, virtualization.
Example: EC2. Most control, most responsibility.
PaaS — Platform as a Service
You manage: apps and data only.
AWS manages: everything else including OS.
Examples: Elastic Beanstalk, RDS.
SaaS — Software as a Service
You manage: data and usage only.
AWS manages: all infrastructure and software.
Examples: WorkMail, Chime.
AWS global infrastructure
Regions
Separate geographic areas, each with multiple AZs. Choose by: latency, compliance, pricing, feature availability.
Availability Zones (AZs)
Physically separate data centers within a Region, typically <100 km apart. Deploy across AZs for HA and fault tolerance.
Edge Locations
CloudFront CDN PoPs worldwide. Cache content close to users. Separate from Regions and AZs.
Local Zones
AWS infrastructure placed in major metros for ultra-low latency. Smaller service catalog than full regions.
Shared responsibility model
AWS — Security OF the cloud
Physical datacenters, global network, hardware, hypervisor.
For managed services: RDS engine patching, Lambda runtime, managed Kubernetes control plane.
Customer — Security IN the cloud
IAM config, OS patching on EC2, Security Groups, NACLs, encryption decisions, application code, data classification and protection.
Line shifts by service type. RDS: AWS patches the DB engine. EC2: you patch the OS. Always identify: managed or unmanaged service?
Cloud economics
CapEx vs OpEx
Traditional IT = upfront capital expenditure (CapEx). Cloud = operational expenditure (OpEx) — pay as you go. Converts fixed costs to variable costs. No need to guess capacity upfront — right-size over time.
TCO
Total Cost of Ownership. On-premises hidden costs: hardware procurement, racking, power, cooling, physical security, network gear, licenses, staff salaries, building lease, refresh cycles every 3–5 years. Cloud TCO often wins when all these are counted — not just compute cost.
ROI
Return on Investment. Cloud delivers ROI through: reduced CapEx, no hardware refresh cycles, elasticity (scale down = immediate savings), speed to market (launch in minutes), and reduced operational overhead (fewer staff for routine maintenance).
Economies of scale
AWS purchases compute, storage, and network at massive scale — millions of servers. Per-unit costs far lower than any individual organization. Savings passed to customers through continued price reductions (~60+ price cuts since launch).
AWS Pricing Calculator
Free tool at calculator.aws to estimate costs before building. Model multiple services and regions. Export as CSV or share link. Useful for migration business cases and TCO comparisons.
AWS Cost Explorer
Visualize historical and forecasted spend. Filter by service, tag, account, region. Identifies cost anomalies. Provides Savings Plans and Reserved Instance purchase recommendations based on actual usage patterns.
AWS global infrastructure — deep dive
Regions
Physically separate geographic areas. 30+ regions globally. Each region is completely independent — data does NOT replicate cross-region automatically. Choose based on: regulatory/data residency requirements, latency to end users, feature availability (new services often launch in us-east-1 first), and pricing (rates vary by region).
Availability Zones
Each region has 2–6 AZs (typically 3). Each AZ is 1+ discrete data centers with redundant power, networking, and connectivity. Physically separated by meaningful distance (km) but connected with low-latency fiber. Deploy across AZs for HA. AZ names (e.g., us-east-1a) are randomly mapped per account — 1a in your account may be different physical AZ than 1a in another account.
Edge Locations (PoPs)
400+ Points of Presence (edge locations + regional edge caches) worldwide. Used by CloudFront CDN, Route 53, AWS Shield, and Global Accelerator. Serve content close to users. Separate from Regions and AZs — there are many more edge locations than regions.
Local Zones
AWS infrastructure extensions placed in major metros (e.g., Los Angeles, Dallas, Chicago). Smaller service catalog than full regions. For workloads needing single-digit millisecond latency to specific cities. Create a subnet in a Local Zone from within your VPC.
Wavelength Zones
AWS compute infrastructure embedded in telecom providers' 5G networks. Ultra-low latency for 5G mobile devices — sub-10ms. Use cases: live video streaming, AR/VR, connected vehicles, real-time gaming. Traffic stays in telecom network — never crosses public internet to reach AWS compute.
AWS Outposts
AWS-managed rack of servers installed in your own data center or colocation facility. Runs the same AWS APIs, tools, and hardware as the cloud. For workloads that must remain on-premises due to latency, data sovereignty, or regulatory requirements. AWS installs, manages, and maintains the hardware.
Well-Architected Framework — pillars deep dive
Operational Excellence
Design principles: perform operations as code (IaC), make frequent small reversible changes, anticipate failure, learn from operations events. Key services: CloudFormation, Systems Manager, CloudWatch, X-Ray. Anti-pattern: manual changes, no runbooks, no post-incident reviews.
Security
Design principles: implement a strong identity foundation (IAM), enable traceability (CloudTrail, Config), apply security at all layers, automate security best practices, protect data in transit and at rest, keep people away from data. Anti-pattern: standing access, no MFA, unencrypted data, no audit logs.
Reliability
Design principles: automatically recover from failure (health checks + ASG), test recovery procedures (chaos engineering), scale horizontally, stop guessing capacity, manage change with automation. Key: Multi-AZ deployments, automated backups, DR planning.
Performance Efficiency
Design principles: democratize advanced technologies (use managed services), go global in minutes, use serverless architectures, experiment more often, consider mechanical sympathy. Pick the right service for the job — don't default to EC2 when Lambda or Fargate is more appropriate.
Cost Optimization
Design principles: implement cloud financial management, adopt a consumption model, measure overall efficiency, stop spending on undifferentiated heavy lifting, analyze and attribute expenditure. Key: right-sizing, savings plans, auto scaling, spot instances, lifecycle policies.
Sustainability
Design principles: understand your impact, establish sustainability goals, maximize utilization, anticipate and adopt new hardware/software offerings, use managed services, reduce downstream impact. Minimize environmental footprint by right-sizing and using efficiency features.

Migration strategies — the 6 Rs
Rehost ("Lift and Shift")
Move applications as-is to EC2 without changes. Fastest migration. No cloud optimization. Use when: time pressure, large server estate, minimal budget for code changes. Tool: AWS Application Migration Service (MGN).
Replatform ("Lift, Tinker, Shift")
Minor optimizations without changing core architecture. Example: move on-premises MySQL to RDS (same code, managed DB). Some cloud benefits without full refactor.
Repurchase ("Drop and Shop")
Replace with a SaaS product. Example: migrate CRM to Salesforce, email to Microsoft 365. Abandon existing license investment — often the right choice for commodity software.
Refactor / Re-architect
Redesign application using cloud-native services. Example: break monolith into microservices on Lambda/Fargate. Most expensive and time-consuming — but highest cloud benefit (scale, cost, agility).
Retire
Decommission applications that are no longer needed. Typically 10–20% of application portfolios can be retired during a migration assessment. Reduces costs immediately.
Retain ("Revisit")
Keep on-premises for now — not ready to migrate. Reasons: recent hardware investment, compliance constraints, dependencies not yet resolved. Revisit later.
// chapter 2 · networking
Virtual Private Cloud (VPC)
VPCs are your private network inside AWS. Subnets, route tables, security groups, NACLs, gateways, and connectivity options. One of the heaviest exam topic areas.
📚 Ch.2 — Virtual Private Cloud
🏋
Exam focus: Stateful vs stateless (SGs vs NACLs). Public vs private subnet placement rules. When to use IGW vs NAT gateway. Direct Connect vs VPN. Interface endpoint vs gateway endpoint. VPC peering limitations vs Transit Gateway.
Core VPC components
VPC
A logically isolated virtual network in AWS. Lives in one Region. You define the IPv4 CIDR block (e.g., 10.0.0.0/16). Resources inside are isolated by default.
Subnets
Subdivisions of a VPC within a single AZ. Public subnet: has a route to an Internet Gateway. Private subnet: no direct internet route. Rule of thumb: web frontends in public, data in private.
Route Tables
Control where subnet traffic is directed. Each subnet is associated with exactly one route table. The main route table applies by default; create custom tables for granular control.
Internet Gateway (IGW)
Allows resources in a public subnet to communicate with the internet. Horizontally scaled, redundant, HA by design. One IGW per VPC.
NAT Gateway
Allows private subnet resources to initiate outbound internet connections (e.g., to download patches) without exposing them to inbound internet traffic. Managed by AWS, deployed in a public subnet.
CIDR notation
Specifies IP ranges. /16 = 65,536 IPs. /24 = 256 IPs. /32 = single IP. AWS reserves 5 IPs per subnet (first 4 + last 1).
Stateful vs stateless — the critical distinction
Security Groups — STATEFUL
Remembers which connections were allowed. Return traffic is automatically permitted. Only need inbound rules for allowed traffic. Operates at the instance level. Default: deny all inbound, allow all outbound.
NACLs — STATELESS
No memory of previous packets. Must explicitly allow BOTH inbound AND outbound rules for a connection to work. Operates at the subnet level. Numbered rules evaluated in order; first match wins.
Classic exam trap: If you add an inbound rule to a NACL allowing port 443, traffic still won't return unless you also add an outbound rule allowing the ephemeral port range (1024–65535).
Common port numbers (exam-tested)
22 — SSH
80 — HTTP
443 — HTTPS
3389 — RDP (Windows)
3306 — MySQL

0.0.0.0/0 = everywhere
Avoid in "most secure" answers.

VPC connectivity options
VPC Peering
Direct connection between two VPCs (same or different accounts/regions). Not transitive — if A peers B and B peers C, A cannot reach C through B. Must create separate peering connections. No bandwidth bottleneck.
AWS Transit Gateway
Hub-and-spoke model connecting many VPCs and on-premises networks. Is transitive — solves the peering mesh problem. Centralized routing. Attach VPCs, VPNs, and Direct Connect gateways.
VPN Connection
Encrypted tunnel over the public internet between on-premises and AWS VPC. Quick to set up (hours). Lower bandwidth. Encrypted by default (IPsec).
AWS Direct Connect
Dedicated private network connection from on-premises to AWS. High bandwidth, consistent latency. Takes weeks to months to provision. NOT encrypted by default — add MACsec or IPsec VPN over DX for encryption. Use when speed/bandwidth is the requirement.
VPC Endpoints (Gateway)
Private connection from your VPC to S3 and DynamoDB only. Traffic stays on the AWS network — never hits the public internet. Free. Uses route table entries.
VPC Endpoints (Interface)
PrivateLink-powered ENI in your subnet for all other AWS services (not S3/DynamoDB). Has an IP address. Charged per hour + per GB. Use when connecting to non-S3/DynamoDB services privately.
Gateway vs Interface endpoint: S3 or DynamoDB → Gateway endpoint (free, route table). Everything else → Interface endpoint (paid, ENI).
Default VPC
AWS creates a default VPC in each region with: a /16 IPv4 CIDR (172.31.0.0/16), a public subnet in each AZ, an attached Internet Gateway, and a default route table routing all traffic to the IGW. Good for quick testing — do not use for production without hardening.
Connect vs Share — a classic distractor [web-sourced]
Transit Gateway — CONNECTS VPCs
Routes traffic between separate VPCs and on-premises networks. Each VPC keeps its own subnets. Use when distinct VPCs need to talk to each other at scale.
AWS RAM — SHARES subnets
Resource Access Manager lets multiple accounts deploy into the same shared VPC/subnets (VPC sharing). Use when teams in different accounts need resources in ONE network. Not the same as connecting separate VPCs.
Exam trap: "Multiple accounts need to launch resources into the same subnets" → AWS RAM / VPC sharing, NOT Transit Gateway. Transit Gateway connects networks; it doesn't share them.
VPC Flow Logs & Gateway Load Balancer [web-sourced]
VPC Flow Logs
Capture metadata about IP traffic to/from network interfaces (source, dest, ports, accept/reject). Publish to CloudWatch Logs or S3. Does not capture packet contents — metadata only. Use for troubleshooting connectivity (e.g., a security group/NACL blocking traffic) and security forensics.
Gateway Load Balancer (GWLB)
A 4th ELB type operating at Layer 3/4. Transparently routes traffic to fleets of third-party virtual appliances (firewalls, IDS/IPS, deep packet inspection). Use when the scenario mentions inserting security appliances inline at scale. Not for normal web traffic (that's ALB/NLB).

AWS reserves 5 IP addresses per subnet — the first four and the last one.

NAT Gateway vs NAT Instance
NAT Gateway — managed (PREFERRED)
Managed by AWS. Starts at 5 Gbps and automatically scales up to 100 Gbps. Deployed in a public subnet; private subnets route 0.0.0.0/0 to it. Charged per hour + per GB. For HA: deploy one NAT Gateway per AZ (separate route tables per AZ). Cannot be used as a bastion host.
NAT Instance — self-managed (LEGACY)
EC2 instance running a NAT AMI. You manage scaling, patching, HA. Must disable source/destination check on the instance. Can double as a bastion host (port 22). Lower bandwidth ceiling. Use only when you need bastion + NAT in one, or when cost is critical.
Exam shortcut: "Highly available outbound internet for private subnets" → NAT Gateway per AZ. "Bastion host that also does NAT" → NAT Instance (rare). "Managed, scalable, no patches" → NAT Gateway.
Security group chaining and reference patterns
SG-to-SG rules
Instead of hardcoding IP CIDRs, reference another security group as the source. Example: allow ALB-SG to reach App-SG on port 8080. When the ALB scales (new IPs), rules still work — no manual IP updates needed. This is the correct architecture for multi-tier applications.
Default SG behavior
Default SG: allows all inbound FROM the same SG (self-referencing), denies all other inbound, allows all outbound. Custom SG you create: denies all inbound, allows all outbound. Never leave default SG on production resources without reviewing rules.
SG rule limits
Default: 60 inbound + 60 outbound rules per SG (soft limit). 5 SGs per network interface. Rules are evaluated collectively — if ANY rule matches, connection is allowed. No rule ordering (unlike NACLs).
NACL rule processing — detailed
Rule numbering
Rules are numbered 1–32766. Evaluated lowest number first. First matching rule wins — no further evaluation. Rule 100 (ALLOW HTTPS) is evaluated before rule 200 (DENY HTTPS). Conventionally use increments of 10–100 to leave room for insertions.
Catch-all rule *
Rule * at the bottom (cannot be edited or removed) = DENY ALL. If no numbered rule matches, rule * catches it and denies. This enforces default-deny on NACLs.
Ephemeral port problem
NACLs are stateless. When a client initiates HTTPS (443), the server responds on an ephemeral port the client opened (typically 1024–65535). You must allow OUTBOUND on ports 1024–65535 from the subnet, or responses won't return. This is the most common NACL misconfiguration.
NACL scope
Applied at the subnet boundary — affects ALL traffic entering/leaving the subnet, not specific instances. One NACL per subnet; one subnet per NACL. Default NACL allows all. Custom NACLs start denying all until you add allows.
VPN and Direct Connect deep dive
Site-to-Site VPN components
Customer Gateway (CGW): your on-premises router/firewall device (or software appliance). Virtual Private Gateway (VGW): the AWS-side VPN endpoint attached to your VPC. Each VPN connection has 2 tunnels (different AZs for HA). Encrypted with IPsec. Bandwidth limited by the CGW hardware (~1.25 Gbps typical max).
Direct Connect virtual interfaces
Private VIF: connect to your VPC private IPs (most common). Public VIF: connect to public AWS service endpoints (S3, DynamoDB) without internet. Transit VIF: connect to a Direct Connect Gateway, which reaches multiple VPCs across regions via Transit Gateway.
Direct Connect Gateway
Connect from one DX connection to VPCs in multiple regions (not just the DX location region). Without DXG, DX only reaches VPCs in the same region as the DX port. Use with Transit Gateway for multi-VPC + multi-region connectivity.
VPN over Direct Connect
DX is not encrypted. Add IPsec VPN tunnel over the DX connection for encryption while keeping the low latency and consistent bandwidth of DX. Best of both: performance + encryption.
AWS PrivateLink vs VPC peering vs Transit Gateway
VPC Peering
Direct 1:1 routing between two VPCs. Full bidirectional IP connectivity. Not transitive. Max 125 peering connections per VPC (mesh becomes unmanageable at scale). Use for small numbers of VPCs needing full connectivity.
Transit Gateway
Hub-and-spoke. Attach thousands of VPCs and on-premises networks. Transitive routing. Centralized routing table. Best for large enterprise multi-VPC environments. Costs more than peering but scales linearly.
AWS PrivateLink
Expose a service (endpoint service) from one VPC to other VPCs via an interface endpoint. One-directional — consumers reach the service but don't get access to the provider's whole VPC. IPs don't need to be non-overlapping. Best for service-to-service access with strict security boundaries.
Bastion host vs AWS Systems Manager Session Manager
Bastion Host (legacy)
EC2 instance in public subnet with port 22/3389 open. Used as a jump box to reach instances in private subnets. Requires: security group rules, key pair management, public IP, VPN or MFA for bastion access. Becomes a single point of attack.
Session Manager (preferred)
No port 22 required. No bastion. No key pairs. SSM Agent on EC2 + IAM role permissions. Access via AWS console or CLI. All session activity logged to CloudTrail and S3/CloudWatch. Revoke access by removing IAM permissions. Audit-friendly and zero attack surface.
Exam tip: Any question asking for "access to EC2 in private subnet WITHOUT opening port 22" or "most secure way to manage EC2 instances" → AWS Systems Manager Session Manager.
Egress-only Internet Gateway (IPv6)
For VPCs using IPv6: resources in private subnets can initiate outbound IPv6 connections to the internet, but inbound connections are blocked. The IPv6 equivalent of a NAT Gateway (but for IPv6, NAT doesn't exist — Egress-only IGW is the solution since IPv6 addresses are public). Does not apply to IPv4 traffic at all.
Network troubleshooting — diagnostic order
Step 1: Check Security Group
Is there an inbound rule allowing the source IP/SG on the correct port? Security groups are stateful — if inbound is allowed, return traffic is automatic. Most connectivity issues are here.
Step 2: Check NACL
Are both inbound AND outbound rules explicitly configured? Remember NACL is stateless — you need both directions. Check ephemeral port range (1024–65535) in outbound rules. Check rule numbering — lower rule may be blocking before higher rule allows.
Step 3: Check route table
Does the subnet's route table have a route for the destination? Public subnet needs a route to IGW for internet traffic. Private subnet needs a route to NAT Gateway for outbound internet. Missing routes = silent black hole (traffic dropped, no error).
Step 4: Check IGW / NAT
Is an IGW attached to the VPC? Is the NAT Gateway in the correct public subnet? Is the NAT Gateway's subnet's route table routing to the IGW? All must be in place for internet connectivity.
VPC Flow Logs
Enable flow logs on the VPC or specific ENI. Look for ACCEPT vs REJECT log entries. A REJECT shows which layer is blocking — SG reject shows in flow log as REJECT on the instance ENI.

IAM — Users, Roles & Policies
IAM is the security backbone of every AWS environment. Every API call is authenticated by an identity and authorized by policies. Master users, groups, roles, policies, and the evaluation logic.
📚 Ch.3 — Identity and Access Management
🏋
Exam focus: Explicit Deny always wins. SCPs restrict but don't grant. Groups can't be principals in resource policies. Roles use STS temporary credentials. Permissions boundaries limit max permissions. Know when to use each STS API call.
Identity types — overview
👤
IAM Users
PERMANENT
Long-term credentials (password + access keys). Best for humans who need console access or scripts needing persistent keys. Max 2 access keys per user for rotation. Prefer roles when possible.
👥
IAM Groups
COLLECTION
Collections of users sharing permissions. Cannot be nested. Users can belong to multiple groups (up to 10).
Groups are NOT identities. Cannot be referenced as a Principal in resource-based policies or trust policies.
🔄
IAM Roles
TEMPORARY
Assumable identities with temporary credentials from STS (15 min – 12 hrs max; configurable per role). No permanent password or access keys. Used by AWS services, cross-account access, and federated users.
🔑
Service-linked Roles
MANAGED
Predefined role tied to one AWS service (e.g., AWSServiceRoleForECS). AWS creates and manages the trust policy. Cannot delete while the service actively uses it.
IAM Users — deep dive
Credential types
Console password — for AWS Management Console login. Access keys (Key ID + Secret) — for CLI/SDK/API calls. MFA device — virtual (Google Authenticator) or hardware. SSH keys — for CodeCommit only. Max 2 access keys per user (enables zero-downtime rotation).
Account limits
Default 5,000 IAM users per account (soft limit, can request increase). Max 10 groups per user. Max 300 groups per account. Max 1,000 roles per account.
Password policy
Account-level password policy controls: minimum length, require uppercase/lowercase/numbers/symbols, password expiry (1–1095 days), prevent password reuse (remember up to 24 passwords), prevent users changing their own password.
Access key rotation
Best practice: rotate every 90 days. Process: create new key → update application → test → deactivate old key → delete old key after confirming. Two keys allow rotation without downtime. Never commit keys to source control.
Root user
The account root user has unrestricted access to everything — cannot be limited by SCPs or policies. Only use for: initial account setup, account closure, support plan changes, enabling MFA Delete on S3, restoring IAM access if all admins are locked out. Enable MFA on root immediately, then never use it day-to-day.
When to use users vs roles
Use users: humans needing console access, legacy applications that can't use IAM roles, CI/CD systems that don't support role assumption. Use roles instead for: EC2/Lambda/ECS workloads (no key management), cross-account access, federated corporate users. Roles = no long-term credentials to rotate, leak, or revoke.
Access key security traps: Never hardcode keys in application code, environment variables in public repos, or EC2 user data. AWS scans public GitHub repos for leaked keys and will notify you (and bad actors will find them first). Always use instance profiles or task roles for compute workloads.
IAM Groups — deep dive
What groups are
A logical collection of IAM users that share the same identity-based policies. When you attach a policy to a group, all members inherit those permissions. Simplifies permission management — change the group policy once instead of updating each user.
Hard limits
Max 300 groups per account. A user can belong to max 10 groups. Groups cannot be nested — no group of groups. Users in multiple groups get the union of all group permissions (explicit denies still always win).
Common patterns
Developers group → EC2/S3/RDS read-write. Ops group → CloudWatch, SSM, CloudFormation. ReadOnly group → AWS managed ReadOnlyAccess policy. Billing group → AWS managed Billing policy. Add/remove users from groups as roles change — never edit individual user policies.
What groups CANNOT do
Cannot be used as a Principal in any policy (resource-based or trust policy). Cannot assume roles. Cannot be members of other groups. Cannot be referenced by ARN in conditions — only individual users and roles can be principals.
Exam pattern: If a question asks "how do you give the same permissions to 50 developers without managing each individually" → IAM Group with a policy attached. If it asks who can be a principal in a resource policy → users and roles only, not groups.
IAM Roles — deep dive
Two policies on every role
Trust policy (who can assume this role) — defines the Principal allowed to call sts:AssumeRole. Permissions policy (what the role can do) — the same identity-based policy structure as users. Both must be in place for a role to work.
Trust policy anatomy
JSON document with Effect: Allow, Action: sts:AssumeRole, and a Principal (AWS account, service like ec2.amazonaws.com, or federated IdP). Example: "Principal": {"Service": "lambda.amazonaws.com"} allows Lambda to assume that role.
EC2 instance roles
Role attached to an EC2 instance via an instance profile. The EC2 IMDS (Instance Metadata Service) automatically vends rotating temporary credentials at 169.254.169.254/latest/meta-data/iam/security-credentials/. SDKs pick these up automatically — no key management needed.
Cross-account roles
Account A creates a role with a trust policy allowing Account B (or a specific user/role in B) to assume it. User in Account B calls sts:AssumeRole with the role ARN from Account A. Gets temporary credentials scoped to Account A's resources. Eliminates need for cross-account IAM users.
Role chaining
When you use role credentials to assume another role, that's role chaining. Session duration is capped at 1 hour regardless of each role's max session duration setting. Avoid chaining in production where possible.
iam:PassRole permission
Required when you assign a role to an AWS service (e.g., creating an EC2 instance with a role, or a Lambda function with an execution role). Without iam:PassRole on the role ARN, the operation fails even if you have EC2 or Lambda permissions. Common exam trap.
Roles vs users — key advantages
No long-term credentials → no risk of key leakage. Credentials auto-rotate (typically every hour). Easy to audit via CloudTrail — every AssumeRole call logged. Revocation is instant — update the trust policy or delete the role. Always use roles for compute workloads.
Service role
Created by you for an AWS service to act on your behalf. Example: CodeDeploy role that can deploy to EC2. Trust principal: the service endpoint (codedeploy.amazonaws.com). You define the permissions policy.
Service-linked role
Pre-defined by AWS for a specific service. Trust policy is managed by AWS. You can view but not edit the trust policy. Cannot delete while the service uses it. Example: AWSServiceRoleForECS for ECS cluster management.
IAM Policies — deep dive
Policy JSON structure
Every policy is a JSON document with: Version (always "2012-10-17"), one or more Statement blocks. Each statement has: Effect (Allow or Deny), Action (API calls like s3:GetObject), Resource (ARN of what the action applies to), and optional Condition block.
Example policy
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["s3:GetObject","s3:PutObject"],
    "Resource": "arn:aws:s3:::my-bucket/*",
    "Condition": {
      "StringEquals": {"aws:RequestedRegion": "us-east-1"}
    }
  }]
}
Action wildcards
s3:* = all S3 actions. s3:Get* = all S3 Get actions. * = every action on every service (AdministratorAccess). Wildcards in Resource: arn:aws:s3:::my-bucket/* = all objects in a bucket. * as Resource = all resources of that type.
Inline vs managed policies
Inline: embedded directly inside one user, group, or role. Deleted when the identity is deleted. Tight 1:1 coupling — use for unique permissions that should never be reused. Managed: standalone policy with an ARN. Can be attached to multiple identities. Two subtypes: AWS managed (Amazon maintains, you can't edit) and customer managed (you own and version).
Policy size limits
Managed policy: 6,144 characters. Inline policy per user: 2,048 characters. Inline policy per role or group: 10,240 characters. Max 10 managed policies attached to a user/role/group (soft limit). If you hit size limits, split into multiple policies or use condition keys to compress rules.
AWS managed policies
Pre-built by AWS. Common ones: AdministratorAccess (full access), ReadOnlyAccess (describe/list everywhere), PowerUserAccess (all except IAM/Organizations), AmazonS3FullAccess, AmazonEC2ReadOnlyAccess. AWS updates them when new services launch. You cannot edit them.

Condition keys — with real examples
aws:SourceIp
Restrict to corporate IP range: "Condition": {"IpAddress": {"aws:SourceIp": "203.0.113.0/24"}}. Use NotIpAddress + Deny to block all other IPs.
aws:RequestedRegion
Limit where resources can be created: "StringEquals": {"aws:RequestedRegion": ["us-east-1","eu-west-1"]}. Useful for data residency requirements.
aws:MultiFactorAuthPresent
Require MFA for sensitive actions: "Bool": {"aws:MultiFactorAuthPresent": "true"}. Apply to Deny statements for S3 DeleteObject or IAM changes when MFA not present.
aws:PrincipalOrgID
Restrict S3 bucket or KMS key to your entire Organization: "StringEquals": {"aws:PrincipalOrgID": "o-xxxxxxxxxxxx"}. Simpler than listing every account ARN.
aws:ResourceTag / aws:RequestTag
Allow actions only on resources with specific tags: "StringEquals": {"aws:ResourceTag/Environment": "Production"}. Enables tag-based access control (TBAC) — users can only touch resources they own.
s3:prefix
Restrict S3 access to a folder: "StringLike": {"s3:prefix": "home/${aws:username}/*"} — each user can only access their own "folder" in a shared bucket using their IAM username as a variable.
STS API calls — when to use each
AssumeRole
Any principal assumes a role. Cross-account and same-account. Duration 15 min – 12 hrs max (role's max session duration setting is 1–12 hrs, configurable).
AssumeRoleWithSAML
SAML 2.0 assertion from enterprise IdP. Corporate SSO into AWS console or API. No permanent AWS credentials for corp users.
AssumeRoleWithWebIdentity
OIDC token from web IdP (Google, Facebook). Use Cognito as the preferred broker for mobile/web apps — don't expose IdP directly to clients.
GetSessionToken
IAM user generates temporary credentials that satisfy aws:MultiFactorAuthPresent conditions. Used when MFA is required for sensitive operations. Duration 15 min – 36 hrs for IAM users (not roles).
GetFederationToken
Long-term credentials for federated users who need temporary access. Duration 15 min – 36 hrs. Used by custom identity brokers. Less common than AssumeRole patterns.
Key IAM concepts
Instance profile
Container that associates an IAM role with an EC2 instance. IMDS vends rotating temporary credentials automatically. Never embed access keys in application code. The SDK credential chain checks instance profile automatically — zero code change needed.
Permissions boundary
Managed policy set on a user or role defining the MAXIMUM permissions it can have. Does NOT grant permissions — only limits them. Effective permissions = identity policy ∩ boundary. Use to let developers create roles without escalating their own privileges.
IAM Policy Simulator
Tool in the AWS console that lets you test what a user/role/group can or cannot do before applying policies in production. Accepts simulated request (service, action, resource) and returns allow/deny with the reason. Essential for debugging complex policy interactions.
IAM Access Analyzer
Identifies resources shared externally (outside your account or organization) and flags overly permissive policies. Enable per-region including rarely-used regions — attackers provision resources in quiet regions. Also generates least-privilege policies from CloudTrail activity.
Credential report
Account-level CSV report showing all IAM users, their credential types, when they last used them, and whether MFA is enabled. Download from IAM console. Useful for auditing unused credentials and enforcing MFA compliance.
IAM Identity Center (SSO)
Centralized workforce SSO for AWS console and multi-account access. Integrates with Active Directory, Okta, Azure AD via SAML/OIDC. Users log in once and get role-based access to assigned accounts. Replaces creating IAM users in every account. Preferred for organizations with many accounts.
Permission escalation trap: If a user can create/modify IAM policies AND attach them to themselves, they can escalate their own permissions to administrator. Always review iam:CreatePolicy, iam:AttachUserPolicy, and iam:PutUserPolicy carefully — these are high-risk permissions.
Amazon Cognito — User Pool vs Identity Pool [web-sourced]
User Pool — AUTHENTICATION (who you are)
Manages sign-up and sign-in for your app's users. Returns JWT tokens proving identity. Supports social login (Google, Facebook, Apple) and SAML/OIDC. Answers: "Is this a valid user?"
Identity Pool — AUTHORIZATION (what you can access)
Exchanges a verified identity (from a User Pool or social IdP) for temporary AWS credentials via STS, so the user can call AWS services (e.g., upload to S3, read DynamoDB). Answers: "What AWS resources can this user touch?"
Exam trap: "Mobile/web app users need to upload directly to S3" → User Pool authenticates them, then an Identity Pool grants the temporary AWS credentials. A User Pool alone cannot grant AWS resource access. Cognito = app/end-user identity; IAM Identity Center (SSO) = workforce/console access.

EC2 instance type families (know the use cases)
General Purpose (M, T)
Balanced CPU/memory/network. T-series = burstable (good for dev/test with low baseline CPU). M-series = consistent general workloads.
Compute Optimized (C)
High CPU ratio. Batch processing, HPC, gaming servers, media encoding, scientific modeling.
Memory Optimized (R, X)
High RAM ratio. In-memory databases, real-time big data analytics, SAP HANA. R = balanced, X = extreme memory.
Storage Optimized (I, D)
High sequential read/write and IOPS. NoSQL databases, data warehouses, distributed file systems.
Accelerated (P, G, Inf)
GPU instances. ML training (P), graphics/gaming (G), ML inference (Inf). High cost.
Auto Scaling Groups (ASGs)
ASGs automatically launch or terminate instances based on demand or schedules. Three capacity numbers: minimum (never go below), maximum (never exceed — important for cost control), desired (target count right now).
Target tracking
Keep a metric (e.g., CPU at 50%) at a target value. AWS automatically adjusts. Simplest to configure — preferred for most use cases.
Step scaling
Scale by specific amounts when alarms breach thresholds. E.g., +2 instances when CPU > 70%, +4 when CPU > 85%.
Scheduled scaling
Pre-provision capacity at known times. E.g., scale up at 8 AM Monday, scale down at 6 PM Friday.
Always set a maximum capacity. Without it, a misconfigured policy or DDoS could spin up thousands of instances.
Containers — ECS vs EKS vs Fargate
Amazon ECS
AWS's native container orchestration. Simpler than Kubernetes. Task definitions describe containers. Services run and maintain a desired count of tasks. Integrates tightly with AWS (IAM, ALB, CloudWatch).
Launch types: EC2 (you manage instances) or Fargate (serverless).
Amazon EKS
Managed Kubernetes control plane. Use when you need Kubernetes compatibility or have existing K8s workloads. More complex than ECS. Supports EC2 and Fargate worker nodes. AWS manages the control plane — you manage node groups.
AWS Fargate
Serverless compute engine for containers. Works with both ECS and EKS. No EC2 instances to provision, patch, or manage. Pay per vCPU/memory per second. Right-size your task resource requests — you pay for what you allocate, not what you use.
When to use Fargate: You don't want to manage EC2 infrastructure, workloads are variable or burst-heavy, or you want to eliminate OS patching responsibility. More expensive per-unit than EC2 but eliminates operational overhead.
Elastic Beanstalk
PaaS that automatically handles deployment, capacity provisioning, load balancing, auto scaling, and health monitoring. You upload your code; Beanstalk handles the rest. Still uses EC2, ELB, ASG under the hood — but you retain full control to modify the underlying resources. Best for developers who want infrastructure simplicity without giving up visibility.
EC2 placement groups
Cluster placement group
All instances placed in the same rack in the same AZ. Lowest possible network latency between instances (10 Gbps+). Required for tightly coupled HPC workloads. Tradeoff: if the rack fails, all instances fail together. Use same instance type for best performance.
Spread placement group
Each instance on a different rack (different hardware). Max 7 instances per AZ per group. Maximizes fault tolerance for critical applications — single hardware failure takes down at most 1 instance. Use for: small number of highly critical VMs.
Partition placement group
Instances spread across logical partitions (each partition = separate rack). Up to 7 partitions per AZ. Hundreds of instances per group. Each partition is aware of which partition it's in. Use for: Hadoop, HBase, Cassandra, Kafka — distributed systems that do their own rack-aware replication.
Spread limit exam trap: Spread placement group = max 7 instances per AZ. If you need 8 instances with maximum HA, spread group is not enough — use partition group or multiple spread groups.
EC2 instance lifecycle and advanced features
User Data
Script run once at instance launch (as root). Used to install software, configure services, pull code from S3. Base64-encoded. Max 16 KB. Access via IMDS at 169.254.169.254/latest/user-data. Does NOT re-run on reboot unless configured to.
Instance Metadata (IMDS)
Data about the instance accessible only from within the instance at 169.254.169.254. Includes: instance ID, region, AZ, IAM role credentials, security groups, hostname. IMDSv2 (preferred): requires session token — prevents SSRF attacks from stealing credentials. Always enforce IMDSv2 in production.
EC2 Hibernate
Saves RAM contents to EBS root volume. Instance stops (no charge), but on restart resumes exactly where it left off (processes resume, network connections maintained). Root EBS must be large enough to hold RAM. EBS must be encrypted. Use for: long-running processes you want to pause, fast boot applications, pre-warmed instances.
Instance store
Ephemeral NVMe storage physically on the host. Fastest possible storage — directly attached hardware. Data lost when instance stops, terminates, or host fails. Persists only through reboots. Use for: scratch data, temp files, buffers, caches — anything you can recreate. Never for persistent data.
Launch templates vs launch configurations
Launch Templates (PREFERRED)
Versioned — can have multiple versions and a default. Supports: Spot + On-Demand in same template, T2/T3 unlimited, dedicated hosts, Capacity Reservations, all instance types. AWS recommends over launch configurations for all new ASGs.
Launch Configurations (LEGACY)
No versioning. Cannot mix Spot and On-Demand. Cannot use T2 unlimited. Cannot specify multiple instance types. Fewer features. AWS recommends migrating to launch templates. Still tested on the exam as a concept.
Dedicated Instances vs Dedicated Hosts
Dedicated Instances
EC2 instances running on hardware dedicated to your account. No other AWS customers on the same physical server. You don't control which specific host you're on. Charged per-instance. Use for: compliance requirements that mandate physical isolation from other tenants.
Dedicated Hosts
A physical server fully dedicated to you. You see and control socket/core/host information. Required for BYOL (Bring Your Own License) software licensed per physical core or socket (Oracle, Windows Server, SQL Server). More expensive. Can be reserved for cost savings.
BYOL = Dedicated Host. License compliance for per-socket/per-core software always points to Dedicated Host, not Dedicated Instance.
Elastic Load Balancing — deep dive
ALB path and host routing
Listener rules evaluate: path (/api/*, /images/*), hostname (api.example.com, www.example.com), query string (?version=2), HTTP headers, source IP. Each rule routes to a specific target group (different EC2 group, Lambda, or IP). Enables true microservices routing from a single LB.
Target groups
Define where traffic is sent: EC2 instances, IP addresses (private), Lambda functions, or other ALBs. Health checks are per target group. One ALB can route to multiple target groups. Targets can span AZs within a region.
Sticky sessions
Duration-based: ALB creates a cookie binding user to a specific target for a set time. Application-based: your app creates a cookie; ALB honors it. Use when session state is stored locally on the instance (preferred: externalize session state to ElastiCache instead — removes need for sticky sessions).
Cross-zone load balancing
ALB: enabled by default (no charge). NLB: disabled by default, charged when enabled. With cross-zone: traffic distributed evenly across ALL instances in ALL AZs. Without: each LB node distributes only within its AZ — if AZs have unequal instance counts, traffic is uneven.
Connection draining (deregistration delay)
When an instance is deregistered or unhealthy, ELB waits (default 300s) for in-flight requests to complete before removing the instance. Prevents sudden connection drops. Set lower (30–60s) for fast-scaling apps. Default 300s can slow deployments unnecessarily.
SSL/TLS termination
ALB and NLB can terminate TLS. Requires an ACM certificate (or imported cert). After termination, backend instances receive HTTP (or you configure re-encryption to backend). ACM handles certificate renewal automatically — no manual cert rotation needed.
NLB static IPs
NLB gets one static IP per AZ (Elastic IP can be assigned). Useful when clients need to whitelist specific IPs (firewalls, on-prem systems). ALB has a DNS name but no static IP — use Global Accelerator if you need a static IP with ALB-style routing.
ELB access logs
Optional — deliver detailed request logs to S3. Include: requester IP, request path, response code, latency, user agent. Useful for security forensics and analyzing request patterns. Enabled per LB; small S3 cost per log.
Auto Scaling — lifecycle hooks and advanced features
Lifecycle hooks
Pause instances at pending (before serving traffic) or terminating (before shutdown). Send notification to SQS or SNS. Use to: install software before instance enters service, drain sessions before instance terminates, capture logs before termination. Default wait: 1 hour.
Warm pools
Pool of pre-initialized EC2 instances (stopped or running) outside the ASG. When scaling out, warm pool instances start instead of cold-launching — dramatically reduces scale-out time. Use when your application has a long initialization/bootstrap time.
Instance refresh
Rolling replacement of all instances in an ASG with new launch template version. Specify minimum healthy percentage (e.g., keep 90% healthy during refresh). Allows automated fleet updates without downtime. Replaced by new instances one batch at a time.
Predictive scaling
Uses ML to forecast future traffic and pre-scale capacity before the load arrives. Looks at 14 days of history to identify patterns. Combines with reactive scaling — predictive handles anticipated load, reactive handles unexpected spikes.
Termination policies
When scaling in, ASG uses termination policy to choose which instances to remove. Default: oldest launch configuration first, then instances closest to billing hour. Always protect instances that should never be terminated (e.g., stateful nodes) using instance scale-in protection.
Spot Instances — interruption handling
How Spot works
Bid for unused EC2 capacity at up to 90% off. AWS can reclaim with a 2-minute notice when capacity is needed. Instance gets a termination notice signal and metadata flag. Design for this by checkpointing work or using Spot-aware frameworks (AWS Batch, EMR, ECS with Spot).
Spot Fleet
Define a mix of instance types and capacities. Fleet maintains target capacity across Spot and On-Demand pools. Automatically diversifies across instance types and AZs to reduce interruption risk. Allocation strategies: lowest price, capacity-optimized (choose pool with most available capacity — best for reducing interruptions).
Spot blocks
Reserve Spot capacity for a specific duration (1–6 hours). Not available to new customers. Prevented interruption during the window. Use for batch jobs with known completion time.
Right workloads for Spot
MUST be: fault-tolerant, stateless or checkpointing, not time-critical. Good fits: big data ETL, ML training, CI/CD build workers, rendering, test environments. Bad fits: production web servers, databases, anything that can't survive sudden termination.
Containers — ECS and EKS deep dive
ECS Task Definition
JSON blueprint defining how containers run: image URI, CPU/memory, port mappings, environment variables, logging configuration, storage mounts. Version-controlled. One task definition can define multiple containers that run together (like a pod).
ECS Task Role vs Execution Role
Task Role: IAM permissions for what your application code can do (call S3, read DynamoDB, publish to SNS). Execution Role: IAM permissions for the ECS agent to run your task (pull image from ECR, write logs to CloudWatch, retrieve Secrets Manager secrets). Both are needed; they're separate for least-privilege.
ECS Service
Maintains a desired count of task replicas. Integrates with ALB (dynamic port mapping). Handles rolling updates (replace tasks one by one) or blue/green deployments. Can span multiple AZs via capacity providers.
EKS Managed Node Groups
AWS provisions and manages EC2 worker nodes — you just specify instance type and count. AWS handles patching, draining on update, and replacement. You still pay for EC2. Simplest way to run EKS without managing node lifecycle manually.
Fargate profile (EKS)
Define which Kubernetes namespaces and labels should run on Fargate (serverless) vs EC2 nodes. Mix Fargate and EC2 in the same cluster. Fargate pods: no nodes to manage, per-pod billing. Trade: less control over the underlying compute.
Amazon ECR
Elastic Container Registry — fully managed Docker/OCI image registry. Private by default. Integrates with ECS, EKS, Lambda container images. Supports image scanning for vulnerabilities (using Inspector). Lifecycle policies to delete old images automatically.

S3, EBS, EFS & FSx
Object, block, and file storage — each has a distinct use case. S3 is the exam's most-tested storage service. Know the storage classes, security controls, replication, and when to use EBS vs EFS vs FSx.
📚 Ch.5 — Storage
🏋
Exam focus: Storage type by use case (object/block/file). S3 storage class tradeoffs. S3 vs EBS vs EFS vs FSx — when to use each. EBS is AZ-scoped (one instance at a time). EFS is multi-AZ, multi-instance. S3 is global/regional, no instance attachment.
Storage types — the fundamental distinction
Object Storage (S3)
Stores data as flat objects (file + metadata + unique key). Massively scalable, no hierarchy. Best for: backups, media, static websites, data lakes, archival. Access via HTTP API.
Block Storage (EBS)
Raw storage volumes that appear as disks to an OS. Attached to a single EC2 instance in the same AZ. Best for: databases, boot volumes, low-latency IOPS workloads.
File Storage (EFS, FSx)
Hierarchical file system accessed via NFS or SMB protocols. Multiple instances can mount the same filesystem simultaneously. Best for: shared workloads, CMS, home directories.
Amazon S3 — core concepts
Buckets & objects
Globally unique bucket names. Objects (files) stored with a key (path). Max object size: 5 TB. Use multipart upload for objects >100 MB. Bucket names are globally unique across all AWS accounts.
Versioning
Enables keeping multiple versions of an object. Protects against accidental deletion and overwrites. Once enabled, can only be suspended — not disabled. Enable before enabling MFA Delete.
MFA Delete
Requires MFA to permanently delete objects or change versioning state. Extra protection for production buckets. Only root user can enable/disable MFA Delete.
S3 Object Lock
WORM (Write Once, Read Many). Governance mode: users with special permission can override. Compliance mode: no one (including root) can delete within retention period. Used for regulatory compliance.
Replication
CRR (Cross-Region Replication) for DR and latency. SRR (Same-Region Replication) for log aggregation or compliance copies. Versioning must be enabled on both buckets.
S3 Website hosting
Host static websites directly from S3. Bucket must be public. Use with CloudFront for HTTPS and global caching. Cannot do server-side processing (use Lambda/API Gateway for dynamic content).
Consistency model
Strong read-after-write consistency for all operations since Dec 2020 — a GET immediately after a PUT or overwrite returns the latest data. Old exam dumps that say "eventually consistent" are outdated.
Durability & availability
11 nines (99.999999999%) durability — objects stored redundantly across multiple devices and AZs. Standard offers 99.99% availability. One Zone-IA stores in a single AZ (lower durability if that AZ is lost).

S3 security controls
Block Public Access
Account-level or bucket-level setting. Overrides bucket policies and ACLs to prevent any public access. Enable by default for all new buckets. Disable only when intentionally hosting a public website.
Bucket policies
Resource-based IAM policies attached to the bucket. Control access from any principal (other accounts, services, public). Use aws:PrincipalOrgID to restrict to your Organization. Use aws:SecureTransport condition to enforce HTTPS-only access (deny HTTP).
Encryption at rest
SSE-S3 (AWS-managed keys, AES-256, default since 2023), SSE-KMS (customer-managed keys, full CloudTrail audit trail per object, fine-grained access control), SSE-C (you supply the key on every request — AWS never stores it). Client-side encryption: you encrypt before upload, AWS never sees plaintext.
Pre-signed URLs
Time-limited URLs granting temporary access to private objects. Generated by an IAM identity using its credentials. Expire after specified time (seconds to days). Use for: time-bounded download access, allowing anonymous upload to a specific key. Signed with the generating identity's permissions — if that identity loses access, the URL still works until expiry.
S3 Access Points
Named network endpoints with their own access policies simplifying access management for large shared buckets. Each access point enforces its own policy. VPC access points restrict access to only within a specific VPC. Use when many applications need different permissions on the same bucket.
S3 Object Ownership
By default (Bucket Owner Enforced), the bucket owner owns all objects even if uploaded by other accounts. ACLs are disabled. Simplifies access management — use bucket policy instead of ACLs. Legacy: object ownership could belong to the uploading account, making access complex.

S3 advanced features
S3 Transfer Acceleration
Uploads route through the nearest CloudFront edge location, then over AWS's backbone to the S3 bucket. Speeds up uploads from geographically distant users. Charged per GB accelerated (only faster when the speed benefit justifies cost — test with the AWS speed comparison tool).
S3 Select and Glacier Select
SQL query to retrieve a subset of data from an S3 object (CSV, JSON, Parquet) without downloading the whole file. Reduces data transfer and processing. Example: SELECT * FROM S3Object WHERE age > 30 on a 10 GB CSV returns only matching rows. Server-side filtering — faster and cheaper than client-side.
S3 Event Notifications
Trigger events on object create, delete, or restore. Destinations: SQS queue, SNS topic, Lambda function, EventBridge. Use for: image processing on upload, virus scanning on new objects, triggering ETL pipelines. EventBridge destination gives the richest filtering and most target options.
Multipart upload
Upload large objects in parts (up to 10,000 parts, 5 MB to 5 GB per part). Allows parallel upload of parts, resuming failed uploads, and uploading before final size is known. Required for objects > 5 GB. Recommended for objects > 100 MB. Incomplete multipart uploads cost money — use lifecycle rules to abort after N days.
S3 Inventory
Scheduled CSV/ORC/Parquet report of object metadata (key, size, last modified, storage class, encryption status, replication status). Delivered to another S3 bucket. Use for compliance auditing, large-scale management, and finding unencrypted objects.
Requester Pays
Requester (not bucket owner) pays for data transfer and request costs. Bucket owner pays only for storage. Use for large public datasets where the owner doesn't want to pay for every download. Requester must be authenticated (anonymous access disabled).
Amazon EBS — block storage for EC2
Scope and movement
AZ-specific. An EBS volume can only attach to an EC2 instance in the same AZ. To move to another AZ: snapshot → copy snapshot → create new volume in target AZ. To share to another region: snapshot → copy snapshot to that region → create volume.
gp3 (General Purpose SSD)
Default volume type. 3,000 IOPS and 125 MiB/s baseline included at no extra cost regardless of size. Can provision up to 80,000 IOPS and 2,000 MiB/s independently (extra cost). 20% cheaper per GiB than gp2. Use for: most workloads, boot volumes, dev/test, small-to-medium databases.
gp2 (legacy General Purpose)
Older default. IOPS tied to volume size (3 IOPS/GiB, max 16,000 IOPS, max 250 MiB/s). Burst up to 3,000 IOPS for small volumes via credit system. Migrate to gp3 — higher IOPS ceiling, higher throughput ceiling, and 20% cheaper. IOPS and throughput cannot be provisioned independently on gp2.
io2 Block Express (Provisioned IOPS)
Up to 256,000 IOPS and 4,000 MB/s throughput. Sub-millisecond latency. 99.999% durability. Use for: Oracle, SQL Server, high-throughput databases needing consistent performance. More expensive — only use when gp3 IOPS limit is insufficient.
st1 (Throughput Optimized HDD)
Low-cost HDD. Max 500 MB/s throughput, 500 IOPS. Cannot be a boot volume. Use for: big data, data warehouses, log processing where sequential throughput matters and cost must be low.
sc1 (Cold HDD)
Lowest cost EBS option. Max 250 MB/s, 250 IOPS. For infrequently accessed sequential data. Cannot be a boot volume. Use when storage cost dominates and performance requirements are minimal.
Snapshots
Point-in-time backups stored in S3 (AWS-managed, not visible in your S3). Incremental — only changed blocks after first snapshot. Can create AMIs from snapshots. EBS Snapshot Archive: cheaper storage for snapshots rarely needed (restore takes 24–72 hrs). Fast Snapshot Restore (FSR): pre-warm snapshot for full performance immediately on restore.
Encryption
Encrypt with AWS KMS key (AWS-managed or customer-managed). All data at rest, in transit between EC2 and EBS, and snapshots are encrypted. Encrypt an existing unencrypted volume: snapshot → create encrypted copy → create new encrypted volume → swap. Cannot directly encrypt an existing unencrypted volume.
Multi-attach
io1/io2 volumes only — attach to multiple EC2 instances in the same AZ simultaneously (up to 16 instances). Application must manage concurrent writes (cluster-aware file systems or distributed lock mechanisms). Not for general use — requires specific application support.
Amazon EFS — deep dive
What it is
Managed NFS (Network File System) file system. Scales automatically (petabyte-scale). Multi-AZ: data stored across multiple AZs in a region. Linux only (POSIX-compliant). Multiple EC2 instances in multiple AZs can mount and read/write simultaneously.
Performance modes
General Purpose (default): lowest latency, ideal for latency-sensitive use cases (web serving, CMS, home directories). Max I/O: higher aggregate throughput but higher per-op latency — use for parallel big data workloads with many clients. Choose at creation, cannot change.
Throughput modes
Bursting: scales with file system size (baseline + burst credits). Provisioned: provision throughput independent of size (when you need more throughput than bursting provides). Elastic (recommended): automatically scales throughput up and down based on workload — simplest, no capacity planning.
Storage classes
Standard: frequently accessed. Standard-IA (Infrequent Access): lower cost, retrieval fee. Lifecycle management automatically moves files not accessed for N days to IA. Use Intelligent-Tiering to automate transitions.
Access points
Application-specific entry points into EFS with enforced POSIX user/group identities and root directory. Simplify access management for containerized applications — each service sees its own root.
FSx variants — four types, four use cases
FSx for Windows File Server
Managed Windows file server. SMB protocol, NTFS, Active Directory authentication and DFS namespaces. Windows ACLs. Supports both Single-AZ and Multi-AZ deployments. Use when: Windows workloads, DFS replication, file shares replacing on-premises Windows file servers.
FSx for Lustre
Parallel file system for HPC and ML. Sub-millisecond latency, hundreds of GB/s throughput. Integrates with S3 — import/export data to S3, process in Lustre. Scratch (no replication, cheapest, temp) vs Persistent (replicated, for longer-term workloads).
FSx for NetApp ONTAP
Fully managed NetApp ONTAP. Multi-protocol: NFS, SMB, and iSCSI simultaneously. Thin provisioning, snapshots, clones, SnapMirror replication. Best for lift-and-shift of existing NetApp workloads to AWS without changing apps or workflows.
FSx for OpenZFS
Managed ZFS file system. NFS protocol. Data compression, snapshots, clones. Up to 1 million IOPS with sub-millisecond latency. Best for Linux workloads that need ZFS capabilities (snapshots, clones) without managing ZFS servers.
FSx protocol decision: "Windows/SMB/AD" → FSx for Windows. "HPC/ML/fastest" → FSx for Lustre. "Multi-protocol NFS+SMB+iSCSI" → FSx for NetApp ONTAP. "Linux NFS + ZFS features" → FSx for OpenZFS. "Multi-AZ/Multi-instance Linux shared storage" → Amazon EFS.
AWS Backup — centralized backup management
🏋
Exam focus: AWS Backup = one service to manage backups across EBS, RDS, Aurora, DynamoDB, EFS, FSx, EC2, S3. Backup vault lock = WORM — even admins can't delete. Cross-account backups = blast-radius isolation. DLM is specifically for EBS snapshots and AMIs on a schedule.
Supported resources
EBS volumes (snapshots), RDS databases (automated snapshots), Aurora clusters, DynamoDB tables, EFS file systems, FSx (Lustre, Windows, NetApp ONTAP, OpenZFS), EC2 instances (EBS-backed AMIs), S3 buckets (continuous backup / PITR), VMware workloads (via AWS Backup gateway on-premises), AWS Storage Gateway volumes. All managed through a single console with consistent policies.
Backup plans
Define when and how to back up resources. A backup plan contains one or more backup rules. Assign resources to the plan by tag (most common — tag resources Backup=Daily and the plan picks them up automatically) or by specific ARN. Resource assignment is dynamic — newly tagged resources are automatically included without updating the plan.
Backup rules
Within a backup plan, each rule specifies: Schedule (cron expression — e.g., daily at 5 AM UTC). Backup window (timeframe during which the backup must start). Retention period (how long to keep the backup before automatic deletion — 1 day to 100 years). Lifecycle (transition backups to cold storage after N days for cost savings). Destination vault (which backup vault stores the recovery points). Copy destinations (other regions/accounts for DR).
Backup vaults
Logical containers that store recovery points (backups). Each vault has an access policy (resource-based policy controlling who can restore, delete, or copy from it). Default vault is created automatically. Create additional vaults for different access control requirements (e.g., a production vault where only the backup service role can write). Encrypted with KMS key — you choose which key per vault.
Backup Vault Lock (WORM)
Applies a Write Once, Read Many (WORM) policy to a backup vault. Once locked: no one can delete recovery points before their retention period expires — not administrators, not the root user, not AWS Support. Two modes: Governance mode (users with special IAM permissions can still remove the lock — for testing). Compliance mode (lock is permanent and irremovable — use for regulatory requirements like HIPAA, PCI DSS, SEC 17a-4). Use for: ransomware protection (attacker with compromised admin credentials cannot delete backups), compliance mandating immutable backups.
Cross-region backup
Configure copy rules in your backup plan to automatically copy recovery points to a different AWS region. Use for: Disaster Recovery — if the primary region is unavailable, restore from the copy in the secondary region. Latency for copy depends on backup size. Charges: data transfer out of source region + storage in destination region.
Cross-account backup
Copy backups to a completely separate AWS account (ideally a dedicated backup account with no workloads). Blast-radius isolation: if the production account is compromised (ransomware, accidental deletion, malicious insider), the attacker cannot access backups in a separate account with different credentials. Requires AWS Organizations — the backup account must be in the same organization. Grant cross-account copy permissions via Organizations policies. Combine with Vault Lock for immutable, blast-radius-isolated backups.
AWS Backup for Organizations
Create backup policies at the Organizations level — apply backup plans to all accounts in an OU automatically. New accounts joining the OU inherit backup policies. Centralized compliance: ensure every account backs up tagged resources without individual account configuration. View backup compliance across all accounts in a single dashboard (Backup audit manager).
Backup Audit Manager
Framework-based compliance checking for backups. Pre-built controls: "RDS instances must be backed up daily," "Backups must be retained for at least 35 days," "Backups must exist in a cross-region copy." Generates compliance reports. Integrates with AWS Config for continuous compliance checking. Use for demonstrating backup compliance to auditors.
Restore testing
AWS Backup Restore testing: automatically restore recovery points on a schedule and validate them. Confirms backups are actually restorable — a backup you've never tested restoring is a backup you can't count on. Generates restore testing reports showing success/failure rates and restore times. Automates the "do we actually have working backups?" compliance question.
Data Lifecycle Manager (DLM) — EBS snapshots and AMIs
What it is
Automates the creation, retention, and deletion of EBS snapshots and AMIs on a schedule. Simpler than AWS Backup for EBS-only scenarios — no vault configuration needed. Create lifecycle policies that: take snapshots daily/weekly, retain N most recent snapshots, copy to other regions, share to other accounts.
Lifecycle policy components
Target resources: specify by tag (e.g., Snapshot=Yes) or instance ID. Schedule: frequency (every 12 hours, daily, weekly), start time (offset from epoch). Retention rules: count-based (keep last N snapshots) or age-based (delete snapshots older than N days). Fast snapshot restore: enable FSR in target AZs so snapshots restore at full performance immediately. Cross-region copy: automatically copy each new snapshot to another region.
DLM vs AWS Backup for EBS
DLM: simpler, EBS snapshots and AMIs only, slightly more granular scheduling options, no vault concept, free (pay only for snapshot storage). AWS Backup for EBS: integrates with all other AWS Backup features — vault lock, cross-account copy, Backup Audit Manager, Organizations policies. Rule of thumb: EBS-only backup automation → DLM is simpler. Multi-service backup compliance, WORM, cross-account → AWS Backup.
AMI lifecycle policies
Automate creation of AMIs (golden images) from running EC2 instances on a schedule. Define retention count — keep last 5 golden images, delete older ones. Combine with EC2 Image Builder for fully automated AMI pipelines: Image Builder creates a hardened/patched AMI, DLM manages retention. Ensures you always have recent, clean AMIs for Auto Scaling Groups.

Route 53 — record types and Alias deep dive
A record
Maps a hostname to an IPv4 address. e.g., example.com → 1.2.3.4. The fundamental DNS record type.
AAAA record
Maps hostname to IPv6 address. Same as A but for IPv6.
CNAME record
Maps hostname to another hostname. Cannot be used on the zone apex (root domain: example.com) — only subdomains (www.example.com). Cannot point to AWS resource DNS names at the apex.
Alias record (AWS-specific)
Maps hostname to an AWS resource DNS name (CloudFront, ALB, S3 website, API Gateway, Elastic IP). Can be used at zone apex. Free — no charge for Alias queries to AWS resources. Automatically follows IP changes when the resource changes. Preferred over CNAME for AWS resources.
ALIAS vs CNAME exam rule
If target is an AWS resource (CloudFront, ALB, S3 bucket website, ELB, API GW, Elastic Beanstalk) → use Alias. If it's at the root domain (example.com not sub.example.com) → must use Alias (CNAME not allowed at zone apex).
Route 53 health checks
Endpoint health checks
Route 53 sends health check probes to your endpoint from multiple global locations. Checks HTTP, HTTPS, or TCP. Endpoint is healthy if it responds with 2xx/3xx and optionally contains a string you specify. Threshold: 3 consecutive failures = unhealthy.
Calculated health checks
Combine the results of multiple child health checks using AND/OR logic. E.g., mark parent healthy only if at least 2 of 3 region health checks are healthy. Used for complex multi-region or multi-component health logic.
CloudWatch alarm health checks
Base the health check on a CloudWatch alarm state. Flexible — can check any CloudWatch metric (DB connections, queue depth, custom app metrics). Use when the endpoint itself isn't directly reachable by Route 53.
Private resources
Route 53 health checkers are on the internet and cannot reach resources in private subnets. Solution: create a CloudWatch metric/alarm for the private resource, then use a CloudWatch alarm-based health check.
Amazon CloudFront — deep dive
Distributions and behaviors
A distribution has one or more cache behaviors — each maps URL patterns (/api/*, /images/*, /*) to different origins and cache settings. Default behavior (/*) is required. This enables routing API calls to API Gateway and static assets to S3 from one CloudFront distribution.
Origin Access Control (OAC)
Secure S3 origins: CloudFront gets access via signed requests, and the S3 bucket policy denies all direct access. Users can only reach S3 content through CloudFront. OAC replaces the older OAI (Origin Access Identity) — OAC supports additional S3 features and all AWS regions.
Signed URLs vs Signed Cookies
Signed URL: one URL grants access to one file. Use for: individual file access, users who don't support cookies (RTMP streaming). Signed Cookie: one cookie grants access to multiple files. Use for: premium subscriber access to a set of files, without changing URLs.
CloudFront Functions
Lightweight JavaScript functions running at the edge. Sub-millisecond execution. For: URL rewrites/redirects, HTTP header manipulation, simple A/B testing, viewer request/response manipulation. Up to 10M requests/second. Does NOT have access to external network or body.
Lambda@Edge
Full Lambda functions running at CloudFront edge locations. Supports Node.js and Python. Can read/write body. Can make external API calls. 4 trigger points: viewer request, origin request, origin response, viewer response. Slower and more expensive than CloudFront Functions — use only when you need the extra capabilities.
Cache invalidation
Force CloudFront to evict cached content before TTL expires. Cost: first 1,000 paths/month free, then $0.005 per path. Use /* to invalidate everything. Alternative: use versioned file names (app.v2.js) — no invalidation needed because it's a new URL.
HTTPS requirements
ACM certificate must be in us-east-1 for CloudFront (global service, always references us-east-1 for certs). Configure: viewer policy (HTTPS only or redirect HTTP→HTTPS) and origin policy (HTTPS to origin) independently.
Global Accelerator vs CloudFront
AWS Global Accelerator
Network-layer (TCP/UDP). No caching. Routes to nearest healthy endpoint over AWS backbone. Provides 2 static Anycast IPs globally. Reduces internet hops. Use for: non-HTTP (gaming, IoT, VoIP), any TCP/UDP app needing static IP, fast failover between regions.
Amazon CloudFront
Application-layer (HTTP/HTTPS). Caches content at edge. Dynamic DNS, no static IP. Use for: web content, APIs, media streaming, anything where caching improves performance. Lambda@Edge and CloudFront Functions for edge logic.
Decision rule: "Static IPs for global app" → Global Accelerator. "Cache content globally" → CloudFront. "Non-HTTP protocol globally" → Global Accelerator. "SSL termination + WAF + CDN" → CloudFront.

RDS, Aurora, DynamoDB & Analytics
The exam's heaviest chapter. RDS managed SQL, Aurora's decoupled architecture, DynamoDB for NoSQL, and analytics services. Know Multi-AZ vs Read Replicas, Aurora Global Database, DynamoDB capacity modes, and when each analytics service applies.
📚 Ch.7 — Data and Analytics
🏋
Exam focus: Multi-AZ = HA/failover (synchronous). Read Replicas = read scaling (asynchronous). Aurora's 6-copy storage across 3 AZs is built-in HA. DynamoDB: on-demand vs provisioned, DAX for caching. ElastiCache: Redis vs Memcached. Kinesis vs SQS for streaming.
Amazon RDS — key concepts
Supported engines
MySQL, PostgreSQL, MariaDB, Oracle, SQL Server, Db2. All run on managed EC2 — AWS handles hardware, OS, and engine patching.
Multi-AZ deployment
Synchronous standby replica in a different AZ. Automatic failover in ~1–2 minutes if primary fails. NOT for read scaling — standby is not accessible until failover occurs. For HA and durability.
Read Replicas
Asynchronous copies for read scaling. Can be in the same region, cross-region, or cross-account. Up to 15 replicas. Can be promoted to standalone DB if needed. Slight replication lag — not for financial transactions requiring consistency.
Automated backups
Daily snapshots + continuous transaction log backups. Enable point-in-time recovery to any second within retention period (1–35 days). Backups stored in S3, managed by AWS.
Patching
AWS automates minor version patching during maintenance windows. Major version upgrades (e.g., MySQL 5.7 → 8.0) must always be done manually — AWS never forces them.
Blue/Green Deployments
Staged switchover for zero-downtime upgrades. Creates a staging environment (Green) with all changes, then cuts over. Can roll back quickly if issues arise.
RDS Proxy
Fully managed database connection pooler. Pools and shares connections so many clients don't exhaust the database. Reduces failover time by up to 66%. Essential for Lambda / serverless apps that open huge numbers of short-lived DB connections. If a scenario says "Lambda overwhelming RDS with connections" → RDS Proxy.
RDS Proxy — deep dive
🏋
Exam focus: Lambda + RDS = connection exhaustion problem → RDS Proxy is the fix. Proxy reduces failover time by up to 66%. IAM database authentication supported. Proxy sits between your app and RDS/Aurora — your app connects to the proxy endpoint, not the DB directly.
The problem it solves
Lambda and RDS connection storms: Every Lambda invocation opens a new database connection. At high concurrency (thousands of Lambda executions), this creates thousands of simultaneous DB connections — far exceeding RDS max_connections (which is limited by instance memory, e.g., a db.t3.micro supports ~66 connections). Connections pile up, queries time out, RDS runs out of memory → database instability. Same problem occurs with any short-lived compute (ECS tasks scaling rapidly, micro-services with frequent cold starts).
How RDS Proxy works
RDS Proxy maintains a warm pool of persistent connections to the actual database. Your application connects to the proxy endpoint (not the RDS endpoint directly). The proxy multiplexes (pins or shares) many client connections to a smaller number of actual database connections. 1,000 Lambda invocations → proxy → maybe 50 real database connections. The database sees manageable load. Proxy is fully managed, Multi-AZ, no server to manage.
Supported engines
MySQL (RDS MySQL + Aurora MySQL), PostgreSQL (RDS PostgreSQL + Aurora PostgreSQL), MariaDB, SQL Server (RDS only). Does NOT support Oracle (as of SAA-C03 exam). Check the supported engine list when choosing Proxy — and remember it for exam scenarios mentioning unsupported engines.
Failover improvement
During an RDS Multi-AZ failover, the proxy automatically reconnects to the new primary. Applications connected to the proxy experience only a brief pause — the proxy handles reconnection transparently. Failover time reduced by up to 66% compared to connecting directly to RDS (where your app must re-resolve the DNS and reconnect). For Aurora Multi-AZ, the proxy also reduces failover impact.
IAM database authentication
RDS Proxy supports IAM authentication — instead of a database username/password, your Lambda function or application generates a temporary IAM authentication token (valid 15 minutes) using its IAM role. The proxy validates this token. Benefits: no hardcoded credentials in Lambda code, credentials rotate automatically, works with Secrets Manager integration. Enable on the proxy — the proxy holds the actual DB credentials in Secrets Manager; clients authenticate to the proxy via IAM.
Secrets Manager integration
RDS Proxy requires credentials to be stored in Secrets Manager (not Parameter Store). The proxy retrieves and caches credentials from Secrets Manager to connect to the database. When Secrets Manager rotates the password, the proxy automatically picks up the new credentials — zero downtime for password rotation. Your application never needs to know the DB password.
Proxy endpoint and connection pinning
Your app connects to the proxy endpoint (not the RDS cluster/instance endpoint). The proxy has a read/write endpoint; Aurora proxies can also have reader endpoints. Connection pinning: some database features (SET statements, temporary tables, prepared statements, stored procedures with session state) require the proxy to pin a client to one specific database connection — reducing multiplexing efficiency. Minimize use of session state to maximize proxy efficiency.
VPC-only — no public access
RDS Proxy is always deployed inside your VPC. It does NOT have a public endpoint. Lambda functions accessing RDS through a Proxy must run in the same VPC (or a peered VPC). This is a common Lambda configuration requirement — enabling VPC access for Lambda and ensuring proper security group rules between Lambda and the Proxy.
When to use RDS Proxy
Use it: Lambda functions connecting to RDS (almost always). Microservices with many short-lived connections. Applications with highly variable load (burst of connections during peak). Need IAM auth for DB access. Want zero-downtime credential rotation. Not needed: Traditional EC2 apps with long-lived connection pools (they manage connections already). Low-connection apps. DynamoDB workloads (DynamoDB has no connection limit issue).
Amazon Aurora
Aurora is a wire-compatible MySQL and PostgreSQL replacement — not a wrapper, but a completely different database engine that speaks MySQL/PostgreSQL. Its architecture decouples the storage layer from the compute layer, enabling unique capabilities:
Storage
6 copies of data across 3 AZs automatically. HA is built-in — no Multi-AZ configuration needed.
Auto-scaling storage
Scales automatically up to 128 TiB (newer engine versions support 256 TiB). Shrinks dynamically when data is removed. No pre-provisioning required.
Read replicas
Up to 15 Aurora Replicas share the same storage volume. Adding replicas doesn't copy data — near-instant scaling.
Aurora Backtrack
MySQL only. Roll database back to any point in time without restoring a snapshot. Fast recovery from accidental deletes. Roll forward and back to find exact restore point.
Aurora Global Database
Primary region for writes, up to 10 secondary read-only regions. <1 second replication lag. Promote a secondary for DR with RPO ~1 second.
Aurora Serverless
Automatically scales compute capacity up and down. Ideal for infrequent or unpredictable workloads. Pay per ACU-second.
Aurora endpoints [web-sourced]
Writer (cluster) endpoint
Always points to the current primary instance for read/write. Automatically follows failover to a new primary — your app keeps using the same endpoint.
Reader endpoint
Load-balances read-only connections across all Aurora Replicas. Point your read traffic here so reads spread automatically as you add/remove replicas.
Custom endpoint
Targets a specific subset of replicas (e.g., route heavy analytical queries to larger instances, keep OLTP reads on others).
Amazon DynamoDB
What it is
Fully managed NoSQL key-value and document database. Single-digit millisecond performance at any scale. No server management. Automatically replicates data across 3 AZs.
Provisioned capacity
You specify read capacity units (RCUs) and write capacity units (WCUs). Use with Auto Scaling. Best for predictable workloads. Can reserve capacity for discounts.
On-demand capacity
Pay per request. No capacity planning. Scales instantly to any throughput. Best for unpredictable or variable workloads. More expensive per request at high scale.
DynamoDB Accelerator (DAX)
In-memory cache for DynamoDB. Microsecond read latency. API-compatible — no application code changes. Use when DynamoDB reads are a bottleneck. NOT for write-heavy workloads (DAX caches reads).
Global Tables
Multi-region, multi-active replication. Write to any region, reads locally. Sub-second replication. Use for globally distributed applications needing low-latency reads and writes worldwide.
DynamoDB Streams
Time-ordered log of item-level changes. Trigger Lambda functions on changes. Use for event-driven architectures, change data capture, audit trails.
Key limits & TTL [web]
Partition key value max 2 KB, sort key max 1 KB, item max 400 KB. TTL auto-deletes expired items at no cost — great for session/temp data. For objects too large for an item, store in S3 and keep the pointer in DynamoDB.
DynamoDB — deep dive
Primary key types
Simple (partition key only): each item uniquely identified by partition key alone. Good for user profiles by user_id. Composite (partition key + sort key): uniqueness requires both keys together. Enables querying by partition key and filtering/sorting by sort key. Example: partition=user_id, sort=timestamp for time-series data.
LSI — Local Secondary Index
Same partition key as table, different sort key. Must be created at table creation time — cannot add later. Shares the table's read/write capacity. Up to 5 LSIs per table. Use when you need to query the same partition with different sort orders.
GSI — Global Secondary Index
Different partition key (and optional sort key) from the table. Can be created at any time. Has its own read/write capacity. Up to 20 GSIs per table. Enables entirely new query patterns on existing data. Example: GSI on email attribute so you can look up users by email (not just user_id).
RCU and WCU calculations
1 RCU = one strongly consistent read per second for items up to 4 KB. Eventually consistent reads use ½ RCU. 1 WCU = one write per second for items up to 1 KB. Transactional reads/writes cost 2x. Example: 10 reads/s on 8 KB items = 10 × (8/4) = 20 RCUs (strongly consistent).
DynamoDB Transactions
TransactWriteItems: write up to 100 items atomically (all succeed or all fail). TransactGetItems: read up to 100 items atomically. Use for: financial transactions, inventory management where partial writes would corrupt data. Cost: 2× the normal capacity units.
Backup options
On-demand backup: full table snapshot at any time, no performance impact. PITR (Point-In-Time Recovery): continuous backups for 35 days, restore to any second in the window. Enable PITR for production tables — it's the only way to recover from accidental writes/deletes during the retention window.
Conditional writes
Write only if an item exists/doesn't exist or an attribute meets a condition. Enables optimistic locking (check version number before updating). Example: only update order if status = "pending" — prevents race conditions in concurrent applications.
Caching — ElastiCache deep dive
Redis
Rich data structures: sorted sets, lists, hashes, pub/sub, geospatial. Supports persistence (RDB snapshots, AOF logs) and replication. Multi-AZ with automatic failover. Cluster mode: sharding across multiple nodes for higher throughput. Use for: leaderboards, session stores, real-time analytics, pub/sub.
Memcached
Simple key-value only. Multi-threaded. No persistence, no replication, no cluster mode. Partitions data across nodes (consistent hashing). Designed for pure horizontal caching. Use when you need simple fast caching at scale without Redis features.
Exam shortcut: persistence / replication / failover / complex data types / pub-sub / geospatial → Redis. Pure caching, multi-threaded, maximum simplicity → Memcached.
ElastiCache caching strategies
Lazy loading (cache-aside)
Read from cache first. Cache miss: read from DB, write to cache, return to client. Cache hit: return cached value directly. Only caches data that's actually requested. Stale data problem: DB updates don't automatically update cache. Add TTL to bound staleness.
Write-through
On every DB write, also write to cache. Cache always has fresh data — no stale reads. Downside: every write is 2 operations (slower writes). Cache may have data that's never read (wasteful). Often combined with lazy loading: write-through for critical data, lazy for everything else.
TTL (Time To Live)
Set expiration on cached items to control staleness. Short TTL = more DB reads, fresh data. Long TTL = fewer DB reads, potentially stale data. Choose TTL based on how much staleness your application can tolerate.
Eviction policies
When memory is full: LRU (Least Recently Used) — evict items not accessed recently. LFU (Least Frequently Used) — evict items accessed least often. allkeys-lru most common. Configure in ElastiCache parameter group.
Kinesis Data Streams — deep dive
Shards
Basic unit of capacity. Each shard: 1 MB/s write, 2 MB/s read per consumer. Data records stored for 24 hours default (extend to 365 days). Add shards (shard splitting) for more capacity; merge shards to reduce cost.
Producers and consumers
Producers push records (using Kinesis SDK, Kinesis Agent, Kinesis Producer Library). Consumers read records. Standard consumers: 2 MB/s shared across all consumers per shard. Enhanced Fan-Out: each consumer gets dedicated 2 MB/s per shard via push (HTTP/2). Use enhanced fan-out when multiple consumers need full throughput.
Ordering
Records within a shard are ordered. All records with the same partition key go to the same shard. Use consistent partition keys for records that must be processed in order (e.g., all events for user_123).
Kinesis vs SQS
Kinesis: real-time streaming, multiple consumers, ordered per shard, replay data within retention window. SQS: async decoupled queue, one consumer processes each message, no ordering (Standard), message deleted after processing. Kinesis for streaming analytics; SQS for task queues.
Kinesis Data Firehose
Serverless — no shards to manage. Buffers and delivers data to S3, Redshift, OpenSearch, Splunk, HTTP endpoints. Near-real-time (60 seconds min latency). Can transform with Lambda before delivery. No replay — once delivered, data is in destination. Use when you need fire-and-forget delivery.

Migrations & Data Transfer
Moving data and applications to AWS. DataSync for file migration, Transfer Family for managed SFTP, Storage Gateway for hybrid access, Snow Family for offline bulk transfer, and DMS for database migration.
📚 Ch.8 — Migrations and Data Transfer
🏋
Exam focus: Match the service to the scenario. Large offline data = Snowball. Ongoing file sync = DataSync. Hybrid NFS/SMB = Storage Gateway. Database migration = DMS. SFTP managed endpoint = Transfer Family. DataSync does NOT support EBS migration.
Storage migration services
AWS DataSync
Automated data transfer from on-premises NFS/SMB/HDFS/S3-compatible to AWS (S3, EFS, FSx). Handles scheduling, encryption, validation, and bandwidth throttling. Does NOT support migration to EBS. Ideal for ongoing synchronization or one-time large migrations over existing network connections.
AWS Transfer Family
Managed SFTP, FTPS, and FTP endpoints backed by S3 or EFS. Lift-and-shift file transfer workflows without changing client software. Authentication via service-managed users, Active Directory, or custom identity providers.
AWS Storage Gateway
Hybrid storage bridging on-premises and cloud. Three modes: File Gateway (NFS/SMB to S3), Volume Gateway (iSCSI block storage backed by S3), Tape Gateway (virtual tape library replacing physical tape systems).
AWS Snow Family — for large-scale offline migration
Snowcone
Smallest device. 8–14 TB. Ruggedized, portable. Can use DataSync to move data online or ship device back. Use for remote/edge locations.
Snowball Edge
Large-scale migration device. Storage Optimized: 210 TB NVMe. Compute Optimized: 28 TB NVMe SSD + 104 vCPUs. Can run EC2-compatible instances locally at the edge. Ship back to AWS for import. Note: no longer available to new customers — existing customers and AWS Partner solutions continue to use it.
Snowmobile
A shipping container on a truck. Up to 100 PB per Snowmobile. For exabyte-scale migrations. AWS sends the truck to your datacenter.
Rule of thumb: If it would take more than a week to transfer data over your internet connection, Snow is more practical than DataSync.
Application & database migration
AWS Application Migration Service (MGN)
Lift-and-shift server migration (replaces the legacy Server Migration Service). Agent-based continuous block-level replication from on-premises to AWS. Once replicated, launch test instances to verify, then cut over with minimal downtime (minutes). Supports physical servers, VMware, Hyper-V, Azure VMs.
AWS Database Migration Service (DMS)
Migrate databases with minimal downtime — source stays operational during migration. Homogeneous: Oracle → Oracle, MySQL → RDS MySQL (same engine). Heterogeneous: Oracle → Aurora PostgreSQL (different engines — use with SCT). DMS replication instance is a managed EC2 that runs the migration. Configure source endpoint + target endpoint + replication task.
Schema Conversion Tool (SCT)
Desktop tool that converts database schema and stored procedures from one engine to another. Oracle/SQL Server → Aurora/PostgreSQL. Provides a report showing what it can auto-convert vs what needs manual effort. Required step before heterogeneous DMS migration.
Application Discovery Service
Discovers on-premises servers and their dependencies before migration. Two modes: Agentless (VMware-based, discovers VMs and utilization from vCenter without agents) and Agent-based (installs a collector on each server for detailed process and network dependency mapping). Feeds data into AWS Migration Hub.
AWS Migration Hub
Central dashboard for tracking migration progress across all services (MGN, DMS, etc.). Groups servers into applications. View migration status from one place. Does not actually migrate anything — it's a tracking layer on top.
VM Import/Export
Import VM images (VMware VMDK, Hyper-V VHD, Citrix XVA) to EC2 AMIs. Export EC2 instances back to VM format. Less automated than MGN — use MGN for ongoing server migrations; VM Import for one-time image conversions or BYOL scenarios.
DataSync — detailed
What DataSync does
Automated data transfer over existing network connections. Handles scheduling, encryption (TLS in transit), data integrity verification (checksums), and bandwidth throttling. Transfers from: on-premises NFS/SMB/HDFS/S3-compatible, other cloud storage → AWS (S3, EFS, FSx). Also transfers between AWS storage services.
DataSync agent
Deploy the DataSync agent (VM) on-premises or in source environment. Agent connects to the DataSync service. Create a task: source location + destination location + schedule + options. First transfer = full copy; subsequent = incremental (only changed files).
When to use DataSync vs Snow
DataSync: existing network connection adequate, ongoing sync needed, network bandwidth allows. Snow: terabytes to petabytes of data where network transfer would take weeks or more, or no reliable internet connection at source site.
Storage Gateway — three modes
File Gateway
Presents an NFS or SMB interface to on-premises applications. Files stored as S3 objects — locally cached copies of recently accessed data for low-latency access. Use: on-premises apps need file system access, but data lives in S3. Seamless S3 integration with file-system semantics.
Volume Gateway
Presents iSCSI block storage volumes to on-premises servers. Two modes: Cached volumes — primary data in S3, frequently accessed data cached locally. Stored volumes — primary data on-premises, asynchronously backed up to S3 as EBS snapshots. Use: on-premises apps need block storage with cloud backup.
Tape Gateway
Virtual Tape Library (VTL) backed by S3 and Glacier. On-premises backup software (Veeam, Veritas, Commvault) writes to virtual tapes. Tapes stored in S3; archive to Glacier for long-term. Seamless replacement for physical tape infrastructure without changing backup software.

Serverless & Application Integration
Lambda, SQS, SNS, EventBridge, and Step Functions form the serverless backbone. Understand Lambda limits, SQS vs SNS patterns, and when Step Functions adds value over direct Lambda chaining.
📚 Ch.9 — Serverless and Application Integration
🏋
Exam focus: Lambda max 15 min — if question mentions long-running, Lambda is wrong. SQS = decoupled queue (pull). SNS = fan-out pub/sub (push). EventBridge = event routing. Step Functions = orchestrate multi-step workflows. Kinesis vs SQS: Kinesis for ordered streaming, SQS for decoupled async processing.
AWS Lambda — limits and cold start
Cold starts
When a Lambda function hasn't been called recently, AWS must provision a new execution environment (download code, start runtime, run init code). Adds 100ms–10s of latency depending on runtime and package size. Java and .NET have longer cold starts than Python/Node.js. Mitigate with: Provisioned Concurrency, smaller packages, ARM64 architecture (SnapStart for Java).
Provisioned Concurrency
Pre-initialize N execution environments so they're always warm. Cold start latency eliminated. Costs more (charged even when idle). Use for: latency-sensitive user-facing APIs where cold starts are unacceptable. Configure on a function version or alias.
Reserved Concurrency
Reserve N concurrent executions exclusively for one function. Two purposes: (1) guarantee capacity for critical functions, (2) throttle a function to protect downstream resources (e.g., limit to 10 concurrent executions so it doesn't overwhelm an RDS database). Setting to 0 effectively disables the function.
Lambda Layers
ZIP archives containing libraries, dependencies, or custom runtimes. Up to 5 layers per function, max 250 MB total unzipped. Share code across functions without including it in every deployment package. Example: pandas/numpy layer shared by all data processing functions.
Lambda container images
Package Lambda function as a container image (up to 10 GB). Use when your deployment package exceeds the 50 MB (zipped) / 250 MB (unzipped) limit, or when you need consistent tooling with your containerized services. Must implement Lambda Runtime Interface Client.
Lambda VPC networking
By default, Lambda runs in an AWS-managed VPC with internet access but no access to your VPC resources (RDS, ElastiCache). Enable VPC access by configuring subnets and SG — Lambda creates an ENI in your VPC. But then Lambda has NO internet access by default — add a NAT Gateway for outbound internet. VPC mode adds cold start latency.
Destinations and DLQ
Async invocations only: Lambda retries failed invocations twice (total 3 attempts). After exhausting retries, send to: Destination (success or failure → SQS, SNS, Lambda, EventBridge) or Dead Letter Queue (SQS or SNS) for failures only. Destinations are more flexible — prefer Destinations over DLQ.
Function URLs
Built-in HTTPS endpoint for a Lambda function. No API Gateway needed. Supports CORS configuration. Authentication: IAM or NONE (for public endpoints). Use for simple webhook receivers or single-function APIs where API Gateway features aren't needed.
Aliases and versions
Publish a version (immutable snapshot) of your function. Create an alias (pointer to a version). Aliases support weighted routing (e.g., 90% → v1, 10% → v2 for canary deployments). API Gateway and Event Source Mappings reference aliases, not versions — enables blue/green deployments.
AWS Lambda — key facts
Max execution time
15 minutes hard limit. If workload exceeds this, Lambda is wrong — use ECS, Fargate, or EC2.
Max memory
10 GB hard limit. More memory = more CPU allocated (proportionally). Also affects cost.
Supported runtimes
Python, Node.js, Java, Go, C#, Ruby, PowerShell. Custom runtimes supported (you manage dependencies).
Concurrency
Multiple invocations run in parallel. Provisioned concurrency keeps functions warm. Reserved concurrency limits max concurrent executions.
Pricing
Pay per invocation + per GB-second of compute. First 1M invocations/month free. Allocate memory carefully — you pay for allocation, not usage.
Triggers
S3 events, API Gateway, DynamoDB Streams, Kinesis, SQS, SNS, EventBridge, Cognito, and more.
Messaging services — SQS vs SNS
Amazon SQS — Queue (Pull)
Decoupled async message queue. Consumers poll for messages. Message stays in queue until processed and deleted.

Standard: At-least-once delivery, best-effort ordering. Highest throughput.
FIFO: Exactly-once, strict order. 300 msg/s (3,000 with batching).

Visibility timeout hides message from other consumers while being processed. Dead-letter queue (DLQ) for failed messages.
Amazon SNS — Topic (Push)
Pub/sub fan-out. One message published to a topic → delivered to ALL subscribers simultaneously.

Subscribers: SQS queues, Lambda, HTTP endpoints, email, SMS, mobile push.

SNS + SQS fan-out pattern: SNS pushes to multiple SQS queues. Each queue processes independently. Standard pattern for fan-out with durability.
SQS limits [web-sourced]: Message size max 1 MB (256 KB is a common myth — actual AWS quota is 1,048,576 bytes; use the Extended Client Library + S3 for payloads beyond that). Visibility timeout up to 12 hours. Message retention 1 minute to 14 days (default 4 days). Long polling (up to 20 s) reduces empty responses and cost vs short polling.
Amazon EventBridge — event buses and rules
🏋
Exam focus: EventBridge is the go-to service for event-driven architectures. Know: default vs custom event buses, event patterns (content-based filtering), Scheduler for cron/rate, Pipes for point-to-point integration, cross-account event routing, and archive/replay.
Default event bus
Every AWS account has one default event bus. AWS services publish their events here automatically — EC2 instance state changes, S3 Object Created notifications routed via EventBridge, ECS task state changes, CloudTrail API events, and hundreds more. You create rules on the default bus to react to AWS service events.
Custom event buses
Create additional event buses for your own applications. Decouple publishers and subscribers — the publisher puts events on the bus; subscribers (rules) pick them up. Enables multi-tenant event routing: one bus per application, domain, or environment. Cross-account: grant another account permission to send events to your bus, or receive events from another account's bus.
Partner event sources
SaaS partners (Zendesk, Shopify, Datadog, PagerDuty, GitHub, etc.) publish events directly to EventBridge. No polling, no webhooks to manage. Accept the partner event source in your account to create a partner event bus. Your rules then react to SaaS events just like AWS events.
Rules
A rule matches events and routes them to one or more targets. Two types: Event pattern rules — filter events by content (JSON pattern matching). Scheduled rules — run on a schedule (cron expression or rate expression like rate(5 minutes)). Each rule can route to up to 5 targets. Rules on the default bus can optionally forward events to another event bus (cross-account or cross-region).
Event patterns — content-based filtering
JSON pattern that must match the event structure for the rule to fire. Supports: exact value match, prefix match, numeric range, OR logic (list of values), AND (multiple fields), NOT, exists/does-not-exist checks. Example — react only when EC2 instance in us-east-1 transitions to stopped: {"source":["aws.ec2"],"detail-type":["EC2 Instance State-change Notification"],"detail":{"state":["stopped"]},"region":["us-east-1"]}. Filtering happens in EventBridge — no Lambda invocations for events you don't care about.
Targets
Where matching events are delivered: Lambda (invoke function), SQS (enqueue message), SNS (publish notification), Step Functions (start state machine), Kinesis Data Streams / Firehose, ECS task, CodeBuild / CodePipeline, API Gateway / HTTP endpoints, another EventBridge bus (cross-account routing). EventBridge retries failed target invocations with exponential backoff. Configure a dead-letter queue (SQS) on the target for events that exhaust all retries.
Input transformation
Transform the event JSON before passing it to the target. Extract specific fields, rename keys, add static values. Example: Lambda receives only {"instanceId": "<$.detail.instance-id>"} instead of the full EC2 event JSON. Reduces payload size and simplifies Lambda code.
Archive and replay
Enable an archive on an event bus to record all events (or filtered events) indefinitely or for a retention period. Replay: re-publish archived events to the same bus as if they just happened. Use cases: debug production issues (replay events to a test Lambda), backfill new consumers (new service joins and needs historical events), disaster recovery testing. Archive storage is charged per GB.
Cross-account events
Scenario A — send to another account: add a resource policy on the target account's event bus granting the source account events:PutEvents. Create a rule in the source account with target = the destination account's bus ARN. Scenario B — forward AWS service events cross-account: use CloudTrail in Organization trail mode + EventBridge organization event buses. Useful for centralized security monitoring (all accounts send events to a security account's bus).
EventBridge global endpoints
Route events to a primary and secondary region bus for high availability. If the primary region is impaired, Route 53 health check fails over event ingestion to the secondary region automatically. Use for multi-region active-active event processing. Extra configuration and cost — only needed for very high-availability event pipelines.
EventBridge Scheduler
What it is
Fully managed scheduler for invoking targets on a schedule — more powerful than EventBridge Scheduled Rules. Supports: rate expressions (rate(1 hour)), cron expressions (UTC), and one-time schedules (invoke once at a specific datetime — something EventBridge Rules don't support). Up to millions of schedules per account.
Targets
Over 270 AWS service API targets — not just Lambda. Invoke Step Functions, ECS tasks, CodeBuild, SQS SendMessage, SNS Publish, Kinesis PutRecord, DynamoDB PutItem, and more directly without a Lambda intermediary. Reduces cost and complexity for scheduled automation.
Flexible time window
Instead of all scheduled jobs firing at exactly the same second (thundering herd), configure a flexible time window (e.g., ±15 minutes). EventBridge Scheduler distributes invocations within that window. Use for non-time-critical batch jobs to avoid spiky load patterns.
vs EventBridge Scheduled Rules
Scheduled Rules = simple cron/rate on an event bus, up to 300 rules, no one-time schedules, goes through event bus. Scheduler = dedicated scheduler service, millions of schedules, one-time schedules, direct API target calls, flexible time windows. Prefer Scheduler for new scheduled automation.
EventBridge Pipes — point-to-point integration
What it is
Point-to-point integration between a source and a target with optional filtering and enrichment in between. Replaces custom "glue" Lambda code that reads from a source, transforms, and writes to a target. Fully managed — no Lambda required for simple integrations.
Pipe components
Source (poll-based): SQS, Kinesis Data Streams, DynamoDB Streams, Kafka (MSK), RabbitMQ. Filter: optional event pattern to drop unwanted events before enrichment. Enrichment (optional): invoke Lambda, Step Functions, API Gateway, or EventBridge API Destination to add data to the event. Target: same 270+ targets as Scheduler — Lambda, SQS, SNS, SFN, ECS, HTTP endpoint, and more.
When to use Pipes
DynamoDB Streams → filter change events → enrich with metadata from Lambda → post to HTTP endpoint. SQS queue → filter by message attribute → directly invoke Step Functions without Lambda. Kinesis stream → filter events → post to third-party API. Use when you previously needed a polling Lambda that just reads, maybe filters, and writes to another service.
Pipes vs Rules
Rules are triggered by events published to an event bus (push model from source). Pipes poll sources like SQS, Kinesis, DynamoDB Streams — these can't publish to an event bus themselves. Pipes = pull sources. Rules = push sources. They complement each other.
EventBridge Schema Registry
What it is
Central registry of event schemas — the shape of events (JSON structure) published to event buses. AWS automatically discovers and registers schemas for all AWS service events. Enable schema discovery on custom buses to auto-detect schemas from your custom events.
Code bindings
From a schema, generate typed code bindings (Java, Python, TypeScript) — classes/interfaces that match the event structure. Download and add to your Lambda — eliminates hand-parsing JSON, enables IDE autocompletion, catches schema mismatches at compile time.
SNS vs EventBridge vs SQS decision:
SNS: Instant fan-out, high throughput, many subscribers simultaneously. Simple routing (topic + filter policies). Email/SMS/push notification delivery. Best for: "notify N things immediately."
EventBridge: Rich content-based filtering. React to AWS service events / SaaS events. Cross-account routing. Archive and replay. Best for: "route events based on what's in them, react to AWS service changes."
SQS: Durable queue, decoupled async processing, one consumer processes each message. Best for: "one service hands work off to another, need durable queue."
AWS Step Functions
Serverless workflow orchestration service. Defines sequences of Lambda functions and AWS service integrations as state machines using Amazon States Language (JSON).

Handles: retries with exponential backoff, error handling, parallel execution branches, wait states, and human approval steps. Use when you need coordination across multiple Lambda functions or services — don't chain Lambda calls directly (hidden state, no visibility into failures).
Standard Workflows: long-running (up to 1 year), exactly-once execution, full audit history. Express Workflows: short-duration (up to 5 min), high-throughput, at-least-once.
Amazon SQS — deep dive
Visibility timeout
When a consumer reads a message, the message is hidden from other consumers for the visibility timeout period (default 30 seconds, max 12 hours). If the consumer processes and deletes the message, it's gone. If not deleted before timeout expires, it reappears for another consumer. Set visibility timeout > max processing time to avoid double-processing.
Dead Letter Queue (DLQ)
After a message fails N times (max receive count exceeded), DLQ receives it instead of the regular queue. Investigate failed messages without blocking queue processing. DLQ must be same type as source (Standard → Standard DLQ, FIFO → FIFO DLQ). Always set a DLQ in production — otherwise messages that always fail loop forever.
Long polling vs short polling
Short polling (default): API returns immediately even if empty. Costs more (empty responses still charged). Long polling: waits up to 20 seconds for a message before returning empty. Reduces empty responses by up to 20x. Reduces cost. Always use long polling (WaitTimeSeconds = 20) unless you need immediate response.
FIFO queue details
Exactly-once processing (deduplication by deduplication ID). Strict ordering by message group ID — messages with same group ID processed in order. 300 transactions/s base (3,000/s with batching). Use for: financial transactions, order processing where sequence matters. FIFO queues end in .fifo.
Message group ID
In FIFO queues, messages with the same group ID are processed in order, one at a time. Different group IDs are processed in parallel. Use to achieve per-customer or per-entity ordering while still scaling across groups.
SNS — message filtering and fan-out
Message filtering
Each SQS or Lambda subscriber can have a filter policy — a JSON document specifying which messages they want. Example: SNS topic receives all orders; a "large-orders" SQS queue only subscribes to messages where orderValue > 1000. Prevents irrelevant messages from reaching a consumer. Up to 5 filter policies per subscription.
SNS FIFO topics
Strict message ordering and exactly-once delivery to SQS FIFO queues. Use for: scenarios requiring ordered fan-out (multiple systems need to process events in the same order). Throughput limited (same as SQS FIFO).
Fan-out pattern
SNS → multiple SQS queues simultaneously. Use when multiple downstream services need to process the same event independently: order system → SNS → [inventory queue, shipping queue, billing queue, analytics queue]. Each processes independently at its own rate.
Amazon API Gateway — deep dive
REST API vs HTTP API vs WebSocket API
REST API: full feature set — API keys, per-client throttling, usage plans, caching, request validation, mock integrations, direct AWS service integrations. HTTP API: simpler, 70% cheaper, lower latency — Lambda and HTTP backends only. No caching, fewer features. WebSocket API: persistent bidirectional connections — real-time apps (chat, live dashboards).
Throttling
Account-level: 10,000 requests/second default (soft limit). Per-stage and per-method throttle overrides. Burst: 5,000 requests. Clients over limit get 429 Too Many Requests. Use usage plans to set per-API-key quotas and throttle limits for customer rate limiting.
Authorization options
Cognito authorizer: validate JWT token from Cognito User Pool. Simple, no custom code. Lambda authorizer (custom): call a Lambda function with the token/headers; function returns IAM policy. Use for: third-party token systems, complex logic. IAM authorization: SigV4-signed requests. Use for service-to-service calls within AWS.
API Gateway caching
Cache responses at the API Gateway level (0.5 GB to 237 GB cache). TTL default 300 seconds. Cache hit = no Lambda invocation = lower cost and latency. Can cache per query string, headers, or path parameters. Invalidate by sending Cache-Control: max-age=0 header.
Direct AWS service integrations
REST API can call AWS services directly (SQS, DynamoDB, SNS, S3) without Lambda — reduces latency and cost. Example: PUT /orders can write directly to DynamoDB without a Lambda function. Called "AWS integration" type (not Lambda proxy).

Security Services
KMS, Secrets Manager, ACM, WAF, Shield, Inspector, GuardDuty, Macie, and Security Hub. The exam tests when to use each — know the purpose of each service without needing deep operational knowledge.
📚 Ch.10 — Security
🏋
Exam focus: KMS for key management and encryption. Secrets Manager for rotating credentials. ACM for TLS certificates. WAF for layer-7 threats (SQL injection, XSS). Shield for DDoS. Inspector = vulnerability scanning. GuardDuty = threat detection via logs. Macie = S3 sensitive data discovery.
Encryption and key management
KMS key types
AWS Owned Keys: created and managed entirely by AWS, used by default for services like DynamoDB. You cannot view, audit, or rotate them. Free. AWS Managed Keys: created by AWS for a service (aws/s3, aws/rds), visible in your account, automatically rotate every year. You cannot delete or modify key policy. Customer Managed Keys (CMK): you create and control. Custom key policy. You control rotation. Most flexibility — required for custom encryption requirements.
AWS KMS
Create and manage cryptographic keys (CMKs). Symmetric (same key encrypt/decrypt) and asymmetric (public/private pair). Automatic key rotation (annual for CMKs). Centralized key policies with IAM integration. Every key operation logged to CloudTrail. Integrates with almost every AWS service for at-rest encryption.
KMS 4 KB limit & envelope encryption [web]
KMS can directly encrypt only up to 4 KB of data per call. For larger data, use envelope encryption: KMS generates a data key, that key encrypts your large data locally, and only the small data key is encrypted by KMS. This is exactly how SSE-KMS works behind the scenes.
KMS multi-Region keys [web]
Share the same key material across regions so data encrypted in one region can be decrypted in another. Standard KMS keys are region-scoped and cannot decrypt each other's ciphertext. Use for multi-region DR and Global Tables encryption.
SSE-S3 vs SSE-KMS vs SSE-C
SSE-S3: AWS manages keys, no audit trail per object. SSE-KMS: You manage keys in KMS, full CloudTrail audit trail, fine-grained access control. SSE-C: You provide keys on every request — AWS never stores them.
AWS Secrets Manager
Store and automatically rotate secrets (database passwords, API keys, credentials). Integrates with RDS for automatic password rotation. Applications retrieve secrets via API — no hardcoded credentials. Charged per secret per month.
AWS Certificate Manager (ACM)
Provision, manage, and deploy SSL/TLS certificates for use with CloudFront, ALB, and API Gateway. Public certificates are free. Automatic renewal. CloudFront certificates must be in us-east-1.
Threat detection and protection
🚧
Amazon Inspector
Automated vulnerability scanning for EC2, Lambda, and container images. Finds OS vulnerabilities and unintended network exposure. Continuous scanning, not one-time. Produces a risk score and recommended remediation.
👁
Amazon GuardDuty
Intelligent threat detection using ML on CloudTrail logs, VPC Flow Logs, and DNS query logs. Detects: compromised instances, reconnaissance, unusual API calls, cryptocurrency mining. Enable per region — no agents needed.
📈
Amazon Macie
Uses ML to discover sensitive data in S3 — PII, credentials, financial data. Creates findings for exposed sensitive data. Useful for data compliance and privacy regulations (GDPR, HIPAA).
🛠
AWS Security Hub
Central security dashboard aggregating findings from GuardDuty, Inspector, Macie, Firewall Manager, and partner tools. Scores against security standards (CIS, PCI DSS). Single pane of glass for multi-account security posture.
CloudHSM vs KMS
AWS KMS
Managed key management. AWS controls the HSM but you control the keys. Multi-tenant (your keys are isolated but on shared hardware). FIPS 140-2 Level 2. No dedicated hardware. Scales automatically. Integrates with all AWS services natively.
AWS CloudHSM
Dedicated physical HSM hardware in your VPC. Single-tenant. FIPS 140-2 Level 3. You manage the HSM users and keys — AWS has no access. No AWS-managed key backup. For: applications with regulatory requirements mandating dedicated HSMs, custom key management software, or off-loading SSL/TLS processing. Higher cost and complexity than KMS.
Exam rule: "FIPS 140-2 Level 3" or "dedicated HSM" or "full control, AWS cannot access keys" → CloudHSM. Everything else → KMS. CloudHSM does NOT natively integrate with S3/RDS encryption — use KMS for service integrations.
Parameter Store vs Secrets Manager
SSM Parameter Store
Free tier available (Standard parameters). Stores configuration data and secrets. Hierarchical naming (/prod/db/password). SecureString type uses KMS encryption. No automatic rotation. Advanced parameters: versioning, expiration notifications. Use for: configuration data, non-rotating secrets, lower cost.
AWS Secrets Manager
Costs ~$0.40/secret/month. Specifically for secrets. Automatic rotation (built-in rotation for RDS, Redshift, DocumentDB; Lambda-based for others). Cross-account secret sharing. Secrets versioned. Use for: database credentials, API keys, anything needing automatic rotation.
Decision rule: "Automatic rotation" → Secrets Manager. "Configuration data / app settings" → Parameter Store. "Cost-sensitive, no rotation needed" → Parameter Store. Both can store encrypted values — key differentiator is automatic rotation.
Amazon Cognito — identity for applications
🏋
Exam focus: User Pools = authentication (who are you? → JWT token). Identity Pools = authorization (give me temporary AWS credentials). The two often work together: User Pool authenticates, Identity Pool exchanges the JWT for AWS credentials. API Gateway integrates natively with User Pools.
Cognito User Pools — Authentication
User directory with sign-up, sign-in, and user management.

• Returns JWT tokens (ID token, access token, refresh token) on successful auth
• Hosted UI: pre-built OAuth2/OIDC login page — your app redirects here, no login form to build
• MFA: TOTP apps (Authenticator), SMS, email OTP
• Password policies, account lockout, compromised-credential check
• Lambda triggers: customize auth flows — pre-signup validation, post-confirmation, custom message, pre/post authentication
• Social federation: Google, Facebook, Apple, Amazon login (OIDC/OAuth2)
• Enterprise federation: SAML 2.0 (Active Directory, Okta, Azure AD)
• User Pool groups for RBAC: assign users to groups → group name appears in JWT → Lambda/API uses group to determine permissions
Cognito Identity Pools — Authorization
Issues temporary AWS credentials (STS AssumeRole) from federated identities.

• Accepts identities from: Cognito User Pool, Google, Facebook, Apple, SAML, OpenID Connect providers, developer-authenticated identities
• Maps identity to an IAM role — authenticated users get one role, unauthenticated (guest) users get another
• Temporary credentials have the permissions of the mapped IAM role
• Use when: mobile/web app needs to call AWS services directly (S3, DynamoDB, Lambda) without going through an API layer
• Unauthenticated access: give guest users limited AWS access (e.g., read-only S3) before they log in
• Role selection: basic (one authenticated role) or rules-based (map different user attributes to different roles for fine-grained control)
User Pools + Identity Pools together
Common pattern: User Pool authenticates the user and issues a JWT. App passes that JWT to an Identity Pool. Identity Pool validates the JWT against the User Pool and returns temporary AWS credentials (via STS). App uses those credentials to call AWS services directly. This separates authentication (who you are) from AWS authorization (what you can do in AWS).
API Gateway + Cognito User Pool
API Gateway has a built-in Cognito User Pool authorizer. Client includes the JWT (access token) in the Authorization header. API Gateway validates the token against the User Pool automatically — no Lambda authorizer needed. If valid, request proceeds. If expired/invalid, API Gateway returns 401. Simple, zero-code auth for REST APIs backed by a User Pool.
ALB + Cognito
ALB can authenticate users with a Cognito User Pool before forwarding requests to targets. Configure an "authenticate-cognito" action on the ALB listener rule. ALB redirects unauthenticated users to the User Pool Hosted UI. After login, ALB adds user claims as HTTP headers (X-Amzn-Oidc-Identity, etc.) to the upstream request. Use to add auth to web apps on EC2/ECS without changing application code.
JWT token types
ID token: contains user identity claims (email, name, group memberships, custom attributes). For the client/app to know who the user is. Access token: used to authorize actions on Cognito APIs (e.g., call UserInfo endpoint, update attributes). Usually what API Gateway validates. Refresh token: long-lived, used to get new ID/access tokens without re-login. Default expiry: ID/access = 1 hour, refresh = 30 days (configurable).
Lambda triggers (exam-relevant)
Pre-signup: validate or auto-confirm sign-ups from trusted email domains. Pre-authentication: block sign-in based on custom logic (device fingerprint, IP block). Post-authentication: log sign-in events to your analytics. Custom message: customize the verification email/SMS content. Post-confirmation: create a user record in DynamoDB when a new user confirms their account. Define/Create/Verify auth challenge: implement custom authentication flows (e.g., magic link, CAPTCHA) beyond username/password + MFA.
Federation — SAML 2.0
Enterprise SSO: configure your corporate IdP (Active Directory Federation Services, Okta, Azure AD, Ping) as a SAML identity provider in the User Pool. Users log in at the corporate IdP → SAML assertion issued → Cognito User Pool accepts it → issues JWT. Your app integrates only with Cognito — not the corporate IdP directly. Single place to manage all federated identity providers.
Adaptive authentication
Cognito analyzes sign-in attempts for risk signals (new device, unusual IP, impossible travel). Automatically adds MFA challenge for risky sign-ins. Returns a risk score. Configure automatic block or require MFA based on risk level. Enabled per User Pool, no code required.
Cognito Sync / AppSync
Cognito Sync (legacy) syncs user data (app preferences, state) across devices via Identity Pool. Not recommended for new apps — AWS replaced it with AppSync for real-time data sync across clients. If exam asks about syncing user profile data across devices: AppSync (GraphQL subscriptions) is the modern answer.
Decision guide: "App needs user login / registration / social login" → User Pool. "App needs to call AWS services (S3, DynamoDB) as the user" → Identity Pool. "API Gateway needs to validate user JWTs" → User Pool Authorizer. "Fine-grained AWS permissions per user" → Identity Pool with rules-based role mapping. Both together = full auth stack for mobile/web apps.
Application protection
AWS WAF — detailed
Web Application Firewall — Layer 7. Protects against: SQL injection, XSS, bot traffic, bad IP ranges, HTTP floods. Attaches to: CloudFront, ALB, API Gateway, AppSync. Rules: custom (you write conditions) or managed rule groups (AWS and third-party — OWASP Top 10, AWS Bot Control, AWSManagedRulesCommonRuleSet). Rate-based rules: auto-block IPs exceeding N requests per 5 minutes.
AWS Firewall Manager
Centrally deploy and manage WAF rules, Shield Advanced protections, Security Groups, and Network Firewall policies across multiple accounts in an Organization. Policies automatically applied to new accounts/resources as they're created. Requires AWS Organizations. Use when you need consistent security controls at scale without per-account configuration.
AWS Network Firewall
Managed stateful firewall at the VPC level. Supports: stateful deep packet inspection, domain-based filtering, protocol detection, intrusion prevention (Suricata-based rules). Deploy in a dedicated subnet. Route traffic through it via route tables. Use for: centralized VPC traffic inspection, blocking specific domains, compliance requirements for IDS/IPS.
AWS Shield Standard
Free, automatic DDoS protection for all AWS customers. Protects against common layer 3/4 DDoS attacks (volumetric, state exhaustion). Applied automatically to CloudFront, Route 53, and ELB. No configuration needed.
AWS Shield Advanced
Paid ($3,000/month + data transfer fees). Enhanced layer 3/4/7 DDoS protection. 24/7 DDoS Response Team (DRT) — call them during an active attack. Financial protection: AWS credits any scaling charges caused by a DDoS event. Advanced attack diagnostics. For: mission-critical applications with strict uptime requirements.

AWS Organizations — account hierarchy and structure
🏋
Exam focus: SCPs restrict, never grant. Management account is NEVER restricted by SCPs. SCPs DO restrict root users in member accounts. Effective permission = IAM permission AND SCP allow (intersection). Consolidated billing pools volume discounts. CloudTrail organization trail = one trail for all accounts.
Management account
The root account that creates the organization. Never restricted by SCPs — it can always do everything regardless of any SCP applied. Use it only for billing and organization management, never for actual workloads. Best practice: separate member accounts for each environment/team; keep management account's blast radius minimal.
Organizational Units (OUs)
Hierarchical containers for accounts. Root OU → child OUs → member accounts. Example hierarchy: Root → Security OU, Infrastructure OU, Workloads OU → (Dev OU, Test OU, Prod OU). OUs inherit SCPs from their parents — a policy at Root OU applies to all accounts everywhere. Max 5 levels of nesting. Accounts can only be in one OU at a time.
Member accounts
Separate AWS accounts managed under the organization. Isolated by design — each has its own VPC, IAM, resources. Receive billing consolidated to management account. Can be created from scratch via Organizations console or invited (existing accounts join with an invitation). Member accounts can be moved between OUs without creating new accounts.
Consolidated billing
All member account charges roll up to the management account. Benefits: (1) Volume discounts pooled — 10 accounts each using 10 TB of S3 counts as 100 TB for pricing tier. (2) Reserved Instance and Savings Plans sharing — unused reservations in one account can offset charges in another. (3) Single invoice for all accounts. Cost allocation tags and Cost Explorer work across the organization.
Delegated administrator
Assign a member account as the delegated administrator for specific services (GuardDuty, Security Hub, Macie, Firewall Manager, etc.). That member account manages the service across the organization without needing management account access. Best practice: use a dedicated Security account as delegated admin for all security services.
CloudTrail organization trail
Create one CloudTrail trail in the management account with "apply to all accounts in the organization" enabled. All member accounts automatically send CloudTrail events to that trail — without any configuration in member accounts. Logs centralized in one S3 bucket in the log-archive account. Member accounts cannot disable or modify this trail. Provides tamper-resistant audit log across the entire organization.
AWS RAM — Resource Access Manager
Share resources across accounts within an organization without creating new copies. Shareable resources: VPC subnets (most common), Transit Gateways, Route 53 Resolver rules, License Manager configurations, CodeBuild projects. VPC subnet sharing: multiple accounts deploy resources into shared subnets in one VPC — simpler than VPC peering. Resources stay in the owner account; other accounts get access to use them. No extra charge for RAM itself.
Service Control Policies (SCPs) — deep dive
What SCPs do
Define the maximum permissions for accounts in an OU. They are guardrails — a ceiling on what an IAM identity in a member account can do, regardless of what the IAM policy grants. Even an account Administrator (AdministratorAccess policy) cannot do something if the SCP denies it. SCPs affect: IAM users, IAM roles, and — critically — the root user of member accounts. They do NOT affect the management account or service-linked roles.
SCPs never grant permissions
An SCP that allows an action does not mean a principal can do it — the principal must also have an IAM policy granting the permission. The effective permission = what IAM allows AND what SCP allows. If either denies or doesn't allow → access denied. Common trap: applying an Allow-all SCP doesn't mean all principals can do everything — IAM still must grant it.
SCP inheritance (cumulative deny)
SCPs are inherited down the OU hierarchy. An account inherits all SCPs attached to the Root, its parent OUs, and itself. SCPs are ANDed together — an action must be allowed by ALL SCPs in the chain. If Root OU has an SCP denying GuardDuty deletion, NO account in the org can delete GuardDuty, regardless of IAM permissions. A Deny in any ancestor SCP always wins.
Allow-list strategy (explicit allows)
Start with a FullAWSAccess SCP at Root (AWS default — allows everything). Then attach additional SCPs that Deny specific actions at the OU/account level. Most flexible — new AWS services are automatically usable unless explicitly denied. Use when: you want to block a few specific things (e.g., deny all actions outside us-east-1, deny disabling CloudTrail).
Deny-list strategy (explicit allows only)
Remove the FullAWSAccess SCP from Root. Replace with SCPs that explicitly Allow only the services each OU needs. All other services are implicitly denied. Maximum restriction — you whitelist exactly what's permitted. Use when: strict compliance requirements dictate that only approved services can be used. More maintenance overhead — add allows when new services are adopted.
SCP JSON structure
Same JSON syntax as IAM policies. Example — deny all actions outside ap-southeast-1: {"Version":"2012-10-17","Statement":[{"Effect":"Deny","NotAction":["sts:AssumeRole","iam:*","cloudfront:*","route53:*","support:*","globalaccelerator:*"],"Resource":"*","Condition":{"StringNotEquals":{"aws:RequestedRegion":"ap-southeast-1"}}}]}. Note: global services (IAM, CloudFront, Route 53) must be excluded from region-restriction SCPs.
Common SCP use cases
Prevent leaving the organization: Deny organizations:LeaveOrganization. Prevent CloudTrail deletion: Deny cloudtrail:StopLogging, cloudtrail:DeleteTrail. Require MFA: Deny all actions when aws:MultiFactorAuthPresent = false (complex — see IAM MFA enforcement). Region restriction: Deny all non-exempt actions where RequestedRegion != allowed region. Block root user: Deny all actions where aws:PrincipalType = Root.
SCPs vs resource policies vs IAM policies
IAM policies: grant permissions to specific principals in the same account. Resource policies: control access to a specific resource (S3 bucket policy, KMS key policy). SCPs: organization-wide guardrails — set ceiling across all accounts. All three are evaluated. For a cross-account request: the resource policy must allow, the caller's IAM policy must allow, AND any SCP must allow. Any deny wins.
AWS Organizations — multi-account best practices
Recommended account structure
Management account: billing and organization management only. Log archive account: receives all CloudTrail logs and Config snapshots from all accounts — immutable, restricted access. Security tooling account: GuardDuty master, Security Hub master, Macie master, AWS Config aggregator. Shared services account: DNS (Route 53 Resolver), shared VPC (AWS RAM), Transit Gateway, CI/CD tools. Workload accounts: one per environment (dev, staging, prod) per application team.
IAM Identity Center (SSO)
Centralized single sign-on for all accounts in the organization. One place to manage who can access which account with what permissions. Integrates with Active Directory or built-in user store. Users get a portal to switch between accounts without separate IAM users per account. Permission Sets = IAM policies assigned to users/groups for specific accounts. Eliminates management of hundreds of IAM users across accounts.
AWS Config aggregator
Aggregate Config compliance data from all accounts and regions into a single account. See organization-wide compliance posture in one dashboard. Identify which accounts have non-compliant resources. Set up in the Security tooling or management account.
AWS Control Tower
What it is
Automated multi-account governance framework. Sets up a landing zone — a pre-configured multi-account environment following AWS best practices. Built on top of Organizations, IAM Identity Center, Config, CloudTrail, and Service Catalog. Removes the complexity of manually setting up all these services per account.
Guardrails
Preventive guardrails: SCPs that block prohibited actions (e.g., "Disallow changes to CloudTrail"). Detective guardrails: AWS Config rules that detect policy violations and alert (e.g., "Detect whether MFA is enabled for root user"). Guardrails are applied per OU. AWS provides ~400 pre-built guardrails across security, operations, and compliance. Some are mandatory (always on); some are elective.
Account Factory
Automated, standardized provisioning of new member accounts. Accounts come pre-configured: baseline networking (VPCs, subnets), mandatory guardrails applied, IAM Identity Center SSO configured, CloudTrail enabled, Config enabled. New team requests an account → Account Factory creates it in minutes, already compliant.

CloudFormation — advanced features
Nested stacks
Break large templates into smaller reusable stacks. A parent stack creates child stacks using AWS::CloudFormation::Stack resource. Pass outputs from child to parent. Use for: common components (VPC, security groups) reused across multiple application stacks. Avoids template size limits (51,200 bytes for inline, 460,800 via S3).
StackSets
Deploy the same CloudFormation template to multiple accounts and/or regions simultaneously. Manage drift and updates from one place. Requires a StackSets administrator account. Use for: deploying baseline infrastructure (Config rules, IAM roles, security services) to all accounts in an Organization.
Custom Resources
Invoke Lambda during stack create/update/delete for any logic that CloudFormation doesn't natively support. Examples: register a domain, call a third-party API, create resources not yet supported in CloudFormation. Lambda receives event with resource properties, signals success/failure.
Change Sets
Preview what changes will happen before applying a stack update. Shows: resources that will be added, modified, or deleted. Always use change sets for production updates — accidentally deleting a database in a stack update is a common incident. Change sets do not execute changes — you review and then execute.
Drift detection
Identify resources that have been manually modified outside CloudFormation. Drift = difference between expected template state and actual resource state. Detect, not remediate — CloudFormation will show what drifted but won't auto-fix it. Re-run the template or import the manual changes.
Stack policies
JSON policies applied to a stack preventing certain resources from being updated or replaced. Example: protect your RDS instance from being accidentally replaced during a stack update. Stack policy applies during stack updates only (not drift detection or imports).
Observability — CloudWatch deep dive
CloudWatch Metrics
Time-series data points. AWS services publish metrics automatically (CPU, NetworkIn, RequestCount, etc.). Custom metrics: your application publishes via PutMetricData API. Metric resolution: Standard (1 minute, default), High-resolution (1 second — extra cost). Metrics retained for 15 months (detail decreases over time).
CloudWatch Alarms
Trigger actions when metric breaches a threshold. States: OK, ALARM, INSUFFICIENT_DATA. Actions: SNS notification, Auto Scaling action, EC2 action (stop/reboot/terminate). Composite alarms: combine multiple alarms with AND/OR logic for alerting only when multiple conditions are true (reduces alert noise).
CloudWatch Logs Insights
Interactive log query language for CloudWatch Logs. Query syntax: fields @message | filter @message like /ERROR/ | stats count(*) by bin(5m). Much faster than searching raw log streams. Use for: troubleshooting, finding error patterns, counting occurrences.
CloudWatch Container Insights
Collect and summarize metrics and logs from ECS, EKS, and Kubernetes. CPU/memory per container, pod, service. Auto-creates dashboards. Enables alerting on container-level metrics without custom instrumentation.
CloudWatch Lambda Insights
Enhanced monitoring for Lambda functions — collects system metrics (CPU, memory, disk, network) and correlates with Lambda invocation metrics. Deployed as a Lambda Layer. Useful for diagnosing performance issues and cold start behavior.
AWS X-Ray — distributed tracing
Traces requests across distributed applications (microservices, Lambda, ECS, EC2). Visualizes the entire request path as a service map. Identifies bottlenecks, latency hotspots, and error sources. Provides: traces (end-to-end path of a request), segments (work done by individual services), annotations (key-value pairs for filtering traces).
Integration
Enable X-Ray daemon on EC2/ECS, or enable for Lambda (no agent needed — just toggle). Requires X-Ray SDK in application code to create segments. SDK automatically traces: outbound HTTP calls, DynamoDB, SQS, SNS, and SQL queries.
Sampling
X-Ray samples requests to avoid overhead. Default: 5% of requests + first request per second. Adjust sampling rules by service, URL, method. High-value transactions: 100% sampling. High-traffic health checks: 0% sampling.

Design Secure Architectures
This is one of the four SAA-C03 exam domains. Pulling together IAM, network security, data security, and encryption into architectural patterns. Know what to apply and when.
📚 Ch.12 — Design Secure Architectures (exam domain)
🏋
Exam focus: Public subnets for web tier, private subnets for data. Direct Connect is not encrypted by default. VPC endpoints for private S3/DynamoDB access. Gateway endpoint for S3/DynamoDB, interface endpoint for everything else. RPO/RTO drive data backup strategy.
Secure IAM access patterns
Principle of least privilege
Grant only the minimum permissions required. Start with no access, add what is needed. Review and remove unused permissions regularly.
RBAC with IAM roles
Define roles based on job functions. Users assume roles rather than having permissions directly. Centralizes permission management. Use STS for cross-account role assumption.
SCPs for multi-account governance
Attach SCPs to OUs to restrict entire account populations. Use Control Tower for automated guardrail deployment across all accounts.
Tag all IAM resources
Attackers rarely tag resources (they need to move fast). Untagged IAM roles, users, or policies are a red flag. Makes anomaly detection much easier.
Secure network architecture patterns
Subnet placement rule
Public subnet: web frontends, load balancers, NAT gateways. Private subnet: application servers, databases, internal services. Never put databases in public subnets.
0.0.0.0/0 caution
Means "anywhere on the internet." In exam answers, if the question asks for the most secure option, avoid answers containing 0.0.0.0/0 in inbound rules.
Direct Connect security
Direct Connect is not encrypted by default. Use MACsec encryption at the physical layer, or create an IPsec VPN tunnel over Direct Connect for encryption.
VPC endpoints
Keep traffic to S3 and DynamoDB within the AWS network using gateway endpoints. Use interface endpoints (PrivateLink) for other services. Eliminates exposure to the public internet.
Data security controls
RPO (Recovery Point Objective)
How much data you can afford to lose. Expressed as time — e.g., RPO of 1 hour means at most 1 hour of data loss. Drives backup frequency: lower RPO = more frequent backups. Near-zero RPO: Aurora Global Database (replication lag <1s), DynamoDB Global Tables, synchronous replication.
RTO (Recovery Time Objective)
How fast you need to be back online after a failure. Low RTO = keep infrastructure warm and running (Warm Standby, Active-Active). High RTO = can restore from cold backup (Backup & Restore — cheaper). Near-zero RTO: Active-Active with Route 53 health checks routing away from failed region in seconds.
Accidental deletion protection
S3: versioning + MFA Delete + S3 Object Lock (WORM). RDS: deletion protection flag (cannot drop DB). DynamoDB: deletion protection flag + PITR. EC2: termination protection flag. Apply ALL layers to production databases — one is not enough.
Encryption at rest
Use AWS KMS for most services. Default keys (AWS managed) for simplicity; customer managed keys (CMK) for audit trail, rotation control, cross-account use. Services encrypting by default: S3 (since Jan 2023), new EBS volumes (account setting), CloudTrail, SQS.
Encryption in transit
TLS enforced using ACM certificates. CloudFront, ALB, API Gateway support HTTPS. Enforce HTTPS-only: S3 bucket policy with aws:SecureTransport deny on HTTP, ALB listener redirects HTTP→HTTPS, CloudFront viewer policy HTTPS only. Certificate renewal is automated in ACM.
Data lifecycle
S3 lifecycle policies automate transition to cheaper storage classes and deletion after retention period. Required for compliance (retain 7 years then delete). Intelligent-Tiering for variable-access data. DynamoDB TTL for automatic item expiration at no cost.
Secure architecture patterns — common exam scenarios
Multi-layer defense for web app
Internet → CloudFront (WAF + Shield + geo restriction) → ALB (security group: allow CloudFront IPs only) → EC2/ECS in private subnet (SG: allow ALB SG only) → RDS in private subnet (SG: allow app SG only). Each layer is a security checkpoint — breach one layer, still stopped by the next.
S3 secure access pattern
S3 bucket: Block Public Access ON, OAC on CloudFront, bucket policy denies all non-CloudFront/non-specific-role access. Objects encrypted with SSE-KMS. Versioning enabled. S3 Object Lock for compliance data. Lifecycle policy to manage versions.
Least privilege at scale
IAM Identity Center (SSO) for all human access — users never have long-term IAM credentials. Roles for all compute. Permission boundaries to delegate role creation safely. SCPs to prevent entire account categories from ever accessing restricted services.
Forensics architecture
GuardDuty detects threat → SNS notification → Lambda isolates EC2 instance (remove from SG, create AMI for forensics, take EBS snapshot) → notify security team. Never terminate the instance before forensics — you lose evidence.

Design Resilient Architectures
High availability, fault tolerance, disaster recovery strategies (RPO/RTO), and scalable loosely-coupled architectures. The four DR strategies are heavily tested — know the cost/RTO/RPO tradeoffs.
📚 Ch.13 — Design Resilient Architectures (exam domain)
🏋
Exam focus: Multi-AZ = HA within a region. Multi-region = DR across regions. The four DR strategies in cost/RTO order. Read replicas for horizontal read scaling. Write bottlenecks require vertical scaling. ElastiCache cuts database read load by up to 80%.

Real HA = deployed across ≥2 AZs (Auto Scaling + ELB, or Multi-AZ database).

Exam tip: If a question mentions "most cost-effective DR" → Backup & Restore. If it mentions "near-zero RTO/RPO" → Active-Active. Pilot light vs warm standby: pilot light has more cold components that need scaling; warm standby is more ready (lower RTO, higher cost).
High availability within a region
Multi-AZ for databases
RDS Multi-AZ: synchronous standby, automatic failover ~1–2 minutes. Aurora: 6 copies across 3 AZs by design. DynamoDB: automatically replicates across 3 AZs. Always deploy databases across multiple AZs.
Auto Scaling + ELB
ELB distributes traffic across instances in multiple AZs. ASG replaces failed instances automatically. Together they handle both load distribution and instance failures.
Read replica scaling
Add read replicas to offload read traffic from primary database. Typical applications have more reads than writes. Does not help with write bottlenecks — for that, vertically scale the primary (bigger instance type).
Scalable, loosely coupled architectures
Three-tier architecture
Web tier (public subnet, ALB + ASG), Application tier (private subnet, ASG), Database tier (private subnet, RDS Multi-AZ). Each tier scales independently. Failure in one tier is isolated. Security groups chain: ALB-SG ← App-SG ← DB-SG (each only allows traffic from the previous tier's SG).
Horizontal scaling
Add more instances. Handled by ASG. Works only for stateless applications. If session state is stored on instance, scaling breaks sessions. Fix: externalize session state to ElastiCache (Redis) or DynamoDB. No sticky sessions needed — any instance can serve any request.
Vertical scaling
Move to a larger instance type. Has an upper limit. Requires instance stop/start (brief downtime unless using Multi-AZ with blue/green). Use for: write-heavy database bottlenecks where horizontal scaling doesn't help (can't scale out writes on a single-primary RDS).
Caching with ElastiCache
Cache frequently read data in Redis or Memcached. Reduce database reads by up to 80%. Common pattern: ALB → App servers → check ElastiCache → on miss, query RDS → write result to cache. Also cache API responses in API Gateway, static assets in CloudFront edge locations.
SQS for decoupling
Place SQS between application tiers. Producer publishes to queue; consumer processes at its own rate. Absorbs traffic spikes without losing messages. Prevents cascading failures if downstream tier is overwhelmed — messages queue up rather than causing timeouts.
Serverless scaling
Lambda scales automatically from 0 to thousands of concurrent executions. No capacity planning. Use when workloads are event-driven and spiky. Not for: long-running tasks (>15 min), steady high-throughput (EC2/ECS cheaper), or applications needing persistent connections.

Design High-Performing Architectures
Selecting the right compute, storage, database, and network services for performance. Instance type selection, caching strategies, database performance optimization, and network acceleration.
📚 Ch.14 — Design High-Performing Architectures (exam domain)
🏋
Exam focus: Match workload characteristics to instance type family. Synchronous = tight coupling, latency-sensitive. Asynchronous (SQS/SNS) = loosely coupled, absorbs spikes. Read replicas + ElastiCache for read-heavy databases. Global Accelerator for global TCP/UDP performance. CloudFront for HTTP caching.
Selecting EC2 instance types for performance
Compute-optimized (C)
High CPU-to-memory ratio. Use for: batch processing, HPC, media transcoding, gaming servers, scientific modeling, ML inference. C6i series is current generation.
Memory-optimized (R, X)
High memory-to-CPU ratio. Use for: in-memory databases (Redis, SAP HANA), real-time big data processing, high-performance relational databases with large working sets.
Storage-optimized (I, D)
High random IOPS or sequential throughput. I4i: NVMe SSDs for NoSQL databases. D3: dense HDD storage for distributed file systems, data warehouses.
Accelerated (P, G, Trn, Inf)
P: ML training (GPU). G: graphics rendering and video encoding. Trn1: purpose-built ML training chips. Inf2: ML inference. Read the question for GPU, ML, or graphics keywords.
Burstable (T)
Baseline CPU with burst credits. Ideal for dev/test, low-baseline workloads with occasional spikes. If credits run out, performance throttles to baseline. Enable unlimited mode to prevent throttling (costs more).
High-performing storage
EBS io2 Block Express
Highest performance block storage. Up to 256,000 IOPS, 4,000 MB/s throughput. Use for: latency-sensitive databases (Oracle, SQL Server, high-IOPS NoSQL).
S3 Transfer Acceleration
Routes S3 uploads through CloudFront edge locations to reduce upload latency from distant locations. Helps when uploading large files from geographically distant users.
FSx for Lustre
Sub-millisecond latency parallel file system. Hundreds of GB/s throughput. For HPC, ML training with massive datasets, financial simulations.
Instance store
Ephemeral NVMe storage physically attached to the host. Fastest possible storage — but data is lost when instance stops. Use for: temp files, scratch space, caches, buffers.
High-performing databases
Read replicas
Offload read traffic. Up to 15 Aurora Replicas share the same storage. Application must route reads to replicas (read endpoint) and writes to primary. Slight replication lag.
ElastiCache for DB caching
Cache query results in memory. Cache hit = microsecond latency instead of milliseconds from DB. Up to 80% read load reduction. Redis for persistence and complex data structures. Application-level cache invalidation logic needed.
DAX for DynamoDB
Fully managed in-memory cache purpose-built for DynamoDB. Microsecond reads (vs milliseconds). API-compatible — no code changes. Only helps with reads; writes still go to DynamoDB.
Aurora Parallel Query
Pushes down analytical query processing to the storage layer. Speeds up complex analytical queries on Aurora MySQL without requiring separate data warehouse.
High-performing networks
AWS Global Accelerator
Routes TCP/UDP traffic through AWS's global network backbone from the nearest edge location. Provides static Anycast IPs (2 IPs globally). Reduces internet hops. Use for: non-HTTP workloads, gaming, IoT, VoIP. Unlike CloudFront, does NOT cache content.
CloudFront
HTTP/HTTPS CDN. Caches content at edge. Use for web applications, APIs, media streaming. Caching means even faster response for repeated requests.
Enhanced Networking
EC2 feature using SR-IOV for higher bandwidth, lower latency, and lower jitter. Uses Elastic Network Adapter (ENA). Required for high-performance compute clusters and HPC workloads.
Placement Groups
Cluster: all instances in same AZ rack — lowest latency between instances (HPC). Spread: instances on different hardware — max HA (7 per AZ). Partition: groups of instances on separate racks — Hadoop/Kafka/Cassandra style workloads.
Microservices and asynchronous patterns
Synchronous: caller waits for response — tight coupling, cascading failures possible. Asynchronous: caller sends message and continues — like putting a letter in a mailbox. Use SQS/SNS for async communication between services. Asynchronous patterns: absorb traffic spikes, prevent cascading failures, allow independent scaling of producers and consumers.
Microservices architecture: Each service independently deployable and scalable. Use ALB with path-based routing to route to different services. ECS or EKS for container-based microservices. API Gateway for managed API surface.
Exam question patterns for Resilient Architectures
"Highly available database"
→ RDS Multi-AZ (synchronous standby, auto-failover). Not Read Replicas — those are for reads, not HA. Aurora: HA built-in with 6 copies across 3 AZs. DynamoDB: automatically multi-AZ. Always specify which engine if asked.
"Application can survive EC2 failure"
→ ALB + ASG across multiple AZs. The ELB detects the unhealthy instance via health checks, stops sending traffic, and the ASG replaces the failed instance. Single instances are NEVER highly available.
"Decouple and absorb spikes"
→ SQS queue between tiers. Producer keeps working even if consumer is overwhelmed. Queue absorbs the burst. Consumer processes at sustainable rate. Classic producer/consumer decoupling pattern.
"Minimize data loss (low RPO)"
→ Synchronous replication (RDS Multi-AZ, Aurora Global), DynamoDB Global Tables, frequent automated backups + PITR. Note: read replicas are asynchronous — slight lag means small data loss possible.
"Session state for auto-scaled EC2"
→ Store sessions in ElastiCache (Redis) or DynamoDB — not on the EC2 instance. Enables true horizontal scaling without session affinity. If sessions must stay on EC2, use ALB sticky sessions (but then scaling out doesn't help all requests).

Design Cost-Optimized Architectures
The fourth SAA-C03 exam domain. Matching the right pricing model to workload patterns, right-sizing compute and databases, minimizing data transfer costs, and using AWS cost management tools effectively.
📚 Ch.15 — Design Cost-Optimized Architectures (exam domain)
🏋
Exam focus: Spot for fault-tolerant batch. Reserved/Savings Plans for steady-state. Auto scaling to avoid over-provisioning. S3 Intelligent-Tiering for unknown access patterns. Serverless for variable workloads (pay only when invoked). Minimize cross-region data transfer. Stop instances when not in use.
Cost-optimized compute
Auto Scaling
Scale down during low demand — don't pay for idle capacity. Scale up only when needed. Set maximum carefully. Always the first cost optimization for variable workloads.
Stop idle instances
EC2 charges only when running. Stop dev/test instances evenings and weekends. Use Lambda functions or EventBridge schedules to automate start/stop. Can save 65%+ on non-production environments.
Right-size instances
Use CloudWatch metrics and Compute Optimizer to identify over-provisioned instances. Downsize to the smallest instance type that meets performance requirements.
Spot for batch
Up to 90% savings for fault-tolerant, stateless, interruptible workloads: ETL, ML training, batch processing, CI/CD, rendering. Design for interruptions — checkpoint frequently.
Serverless for variable loads
Lambda: pay only per invocation. Zero cost when idle. Ideal for event-driven, spiky, or unpredictable workloads. More expensive than EC2 at sustained high throughput — use EC2/ECS/Fargate for steady high-volume.
Cost-optimized storage
S3 storage classes
Move infrequently accessed data to cheaper tiers. Automate with lifecycle policies. Standard-IA for occasional access. Glacier for archival. Intelligent-Tiering when access patterns are unknown — it automatically optimizes.
S3 lifecycle policies
Automate transitions between storage classes and deletion after retention period. E.g., move to Standard-IA after 30 days, Glacier after 90 days, delete after 7 years. Required for compliance and cost management.
EBS right-sizing
Only pay for provisioned capacity on EBS — right-size volumes. Use gp3 (not gp2) — gp3 allows independent scaling of IOPS and throughput without paying for more storage. Delete unattached EBS volumes.
Cost-optimized databases
RDS right-sizing
Use Compute Optimizer and CloudWatch to identify under-utilized RDS instances. Scale down instance type. Use Aurora Serverless for unpredictable database workloads.
Reserved RDS
1 or 3-year RI for steady-state databases. Up to 69% savings over On-Demand. Commit to instance family and region, not specific instance size (with size-flexible RIs).
DynamoDB on-demand
On-demand for unpredictable workloads. Provisioned + auto-scaling for predictable workloads. On-demand is more expensive per request but eliminates capacity planning risk.
Backup retention
Don't retain backups longer than necessary. Shorter retention periods reduce S3 storage costs. Balance compliance requirements against storage cost.
Cost-optimized networking
Data transfer pricing
Data transfer IN to AWS: free. Data transfer OUT to internet: charged (~$0.09/GB first 10 TB/month). Data transfer between AZs in same region: ~$0.01/GB each direction. Between regions: ~$0.02/GB. Same AZ (same instance type): free. Design to co-locate services in same AZ where latency/throughput is important.
Minimize data transfer
Keep S3 bucket and EC2 in same region. Use EFS in same AZ as EC2 (no cross-AZ EFS charge). Use S3 gateway endpoint to avoid NAT Gateway charges on S3 traffic. VPC endpoints eliminate internet egress path (and associated NAT charges) for S3/DynamoDB.
CloudFront for egress reduction
CloudFront caches content at edge. Repeated requests served from cache — no egress charges from origin. CloudFront-to-internet egress is significantly cheaper than EC2/S3 direct egress. For large media files served repeatedly, CloudFront can reduce costs by 80%+.
VPC endpoint cost savings
Gateway endpoints (S3, DynamoDB) are free — eliminate NAT Gateway data processing charges ($0.045/GB through NAT GW) for traffic to those services. If your EC2 instances send significant data to S3, a gateway endpoint pays for itself immediately. Interface endpoints have hourly + per-GB charges — model before deploying.
